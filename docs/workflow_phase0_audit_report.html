4. Phase 1（WF1）前置风险

高风险

  • 节点 handler 依赖未就绪 — 业务节点（bank_import、report 等）需要调用已有的 service 函数，但部分 service 仍依赖 artifact_runtime.run_parser（断链）。如果 WF1 要实现 data.bank_import 节点，必须先解决或绕过 artifact_runtime 断链。
  • 同步执行阻塞风险 — 当前执行引擎是同步阻塞的。如果节点 handler 执行时间长（如大文件解析），会阻塞 HTTP 请求。WF1 需评估是否需要改为异步执行。

中风险

  • 节点间数据传递 schema 未定义 — previous_outputs 传递的是自由格式 dict，节点之间没有约定 schema。如果节点 A 的输出格式变更，节点 B 可能静默失败。WF1 需为每个节点定义明确的 output schema。
  • graph_json 无版本迁移策略 — 如果节点 params schema 变更，旧版本的 graph_json 中的参数可能不再有效。需要向前兼容策略或版本适配层。

低风险

  • 测试覆盖不够细 — 11 个测试覆盖了核心路径，但缺少边界条件测试（空 graph、超大 graph、并发执行等）。
  • 无前端依赖安装 — @vue-flow/core 尚未安装，WF2 阶段需要评估版本兼容性。

5. 下一阶段执行顺序

1
WF1-A：定义节点 output schema
为每个候选节点定义明确的 input/output schema（作为 NodeType dataclass 或 Pydantic 模型）。这是所有后续工作的基础。
2
WF1-B：实现无外部依赖的业务节点
优先实现不依赖 artifact_runtime 的节点（control.start、control.end、data.base_query、report.daily_fund、export.excel）。这些节点可直接调用现有可用 service。
3
WF1-C：增加运行恢复/取消端点
在 workflow_executor 和 workflow_service 中增加 resume/cancel 逻辑，新增 2 个 API 端点。
4
WF1-D：实现 Agent 工具
创建 workflow_ops.py，注册 5 个工具到 permission/tool_registry，让 Agent 能创建和管理工工作流。
5
WF2-A：前端 API 文件
创建 frontend/src/api/workflow.js，对接 13 个端点（11 + resume + cancel）。
6
WF2-B：Vue Flow 画布编辑器
安装 @vue-flow/core，创建 WorkflowEditor.vue + 节点面板 + 属性面板。
7
WF2-C：列表页 + 详情页 + 路由 + 导航
WorkflowList.vue、WorkflowRunDetail.vue、router 更新、MainLayout 导航入口。
