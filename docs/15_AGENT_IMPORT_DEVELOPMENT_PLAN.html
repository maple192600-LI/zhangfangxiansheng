Agent 定位与银行导入后续开发计划

给非开发者看的项目治理说明：Agent 是副驾驶，解析器负责读文件，规则负责填报表。当前最重要的是把上传、预览、确认、入账、报表这条闭环做稳。

一句话结论：继续推动 Agent，但只让它起草方案，不让它替你做财务决定。任何数据进入正式账之前，都必须经过预览、校验和用户确认。

Agent 的定位

Agent 的位置是“副驾驶”，不是“老板”。它可以看样本、起草读取方案、解释处理理由、把候选方案交给系统试跑。

它不能绕过确认入账，不能猜单位和账户，不能自己批准自己生成的方案，也不能变成很多个专用助手来绕开现有架构。

判断 Agent 是否够用，只看三件事：能不能减少重复规则编写，结果能不能试跑和退回，能不能守住用户确认。

解析器和规则的分工

名称大白话解释发生时间保存位置确认方式
解析器 读懂上传文件的说明书，把银行流水或手工 Excel 的行列读成系统认识的数据。 上传文件之后，进入预览之前。 银行流水规则页对应的后端保存表。 用户看试跑结果后保存启用。
规则 填报表的说明书，把已经确认的数据填进指定报表模板。 数据确认之后，生成报表时。 报表模板相关的后端保存表。 用户看模板填写结果后保存启用。

解析器只读文件，不决定最终账户归属。账户归属要靠账号、户名、银行名、开户行、单位名和主数据共同判断。规则只填报表，不重新解析原始银行文件。

银行流水闭环

1. 主数据维护单位、银行、账户、别名。
2. 上传文件用户上传银行流水。
3. 提取线索系统读取账号、户名、银行名、开户行、表头。
4. 匹配解析器找到能读这种文件格式的说明书。
5. 缺失时生成去银行流水规则页上传样本，Agent 起草，系统试跑。
6. 解析预览文件被读成可检查的数据表。
7. 归属匹配系统根据主数据判断单位和账户。
8. 用户确认异常必须修正后才能提交。
9. 正式数据确认后写入资金数据，再生成报表。
不可省略：先有主数据，先预览再入账，没有线索必须让用户处理。

后续开发顺序

第一阶段：治理污染和口径统一

清掉会误导 AI Coding 工具的旧阶段说法，关闭或改写 GitHub 上仍开放的误导项，明确当前只追真实资金日报闭环。

验收：active docs、测试、样本、脚本、协作任务目录搜索不到旧阶段编号表达；剩余命中只允许是外部接口地址、模型商品名、数据库历史迁移文件名或物理表名。

第二阶段：上传结果预览表格可用化

让预览表格像财务人熟悉的 Excel 一样能快速修正数据，支持编辑、复制粘贴、键盘移动、异常筛选和重新校验。

验收：异常未处理时不能提交；多行粘贴不乱位；修改后校验结果能更新。

第三阶段：银行流水识别证据可视化

展示系统从文件里提取了什么线索、匹配了哪些主数据、为什么判断为已匹配、有歧义或未匹配。

验收：用户能看懂匹配理由；多个候选账户时系统不自动选；用户修正能沉淀为以后复用的线索。

第四阶段：解析器生成流程稳定

银行流水规则页负责维护解析器。上传样本后，Agent 起草读取方案，系统试跑，用户确认后保存启用。

验收：同一银行同一格式的不同账户不需要重复写多套方案；试跑为空、字段错位、金额方向错误时不能保存。

第五阶段：报表填写规则执行

让已经确认的资金数据可以填进真实报表模板，并能预览或下载填好的报表。

验收：报表数字能追溯到正式资金数据；模板字段缺失时给出明确错误；失败不会污染正式数据。

第六阶段：整体闭环验收

用多银行、多账户、多单位样本证明系统不是演示，而是能承接日常工作。

验收：从上传银行流水到生成报表，全程不需要改代码；错误数据不会绕过预览进入正式资金数据；同类文件第二次上传更省人工。

给 AI Coding 工具的任务包

目标围绕上传银行流水、预览校验、用户确认、正式资金数据、报表生成闭环开发，不新增平行实现。
必须读取docs/README.md、项目状态、产品范围、数据生命周期、前后端地图、银行导入通用化和本文档。
先确认事实页面是否存在，API 是否被调用，表字段是否真实存在，测试是否覆盖，文档是否与代码冲突。
允许修改当前任务直接相关的页面、API、service、测试、样本和文档。
禁止修改不新增专用 Agent，不新增平行上传或报表链路，不跳过预览，不写死单位、账户、银行格式、金额方向。
测试方式后端单元测试、前端浏览器测试、样本测试、搜索测试。
回滚方案前后端按文件回退；数据库改动必须有迁移回滚或只读兼容策略；测试数据必须说明清理方式。
你只需要抓住四个验收问题： 我上传真实文件了吗？我看见识别和匹配理由了吗？我能改、能确认、能阻止错误提交吗？我最终拿到的报表能追溯到确认过的数据吗？
