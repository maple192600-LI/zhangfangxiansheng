# V1 执行文档 01｜主数据中心与账户初始化

## 一、文档目的

本文档只服务 V1 的第一块正式开发任务。

目标是让 Cloud Code 先把“主数据中心”做成，包含大区或板块、法人单位、银行账户、账户初始化、期初余额与余额日期这条闭环。

这份文档只讨论这一块，不讨论银行流水导入、不讨论手工录入、不讨论日报生成、不讨论发票、不讨论费用审批。

## 二、本块任务在 V1 中的位置

这一块属于 V1 的地基层。

没有主数据中心，后面的导入流水、手工录入、区间日报、筛选导出、数据看板都没有稳定挂载点。

因此本块必须先做，而且必须一次把结构想稳。

## 三、本块唯一目标

用户进入系统后，可以在全中文界面中完成下面这些事情。

先维护大区或板块。

再维护法人单位。

再维护银行账户或其他资金载体账户。

为每个账户录入期初余额与余额日期。

能编辑、停用、查询这些主数据。

能保证后续所有交易、日报、看板、导出都能挂到这些主数据上。

## 四、当前块的业务边界

这一块只负责“主数据”和“初始化”。

它要做的是：

把组织层级搭起来。

把账户台账搭起来。

把第一笔可承接的余额起点搭起来。

它现在不负责：

解析银行流水。

识别新模板。

生成资金日报。

做图表统计。

做 Agent 对话。

## 五、必须落地的数据对象

### 1. 大区或板块 divisions

这是最上层的可选分组。

用户可按自己企业习惯自定义，比如华北大区、华中大区、抖音事业部、淘宝事业部、工程板块、贸易板块。

小公司可以不用，大公司可以启用。

字段最少应包含：

- id
- name
- sort_order
- created_at
- updated_at

### 2. 法人单位 entities

这是正式业务主体。

字段最少应包含：

- id
- division_id，可空
- name
- short_name
- status
- created_at
- updated_at

要求前端显示中文状态，例如“启用”“停用”。

### 3. 银行账户或资金载体 accounts

这是主数据中心里最重要的一张表。

字段必须包含：

- id
- entity_id
- bank_name
- branch_name
- account_number
- account_alias
- account_type
- instrument_type
- input_method
- currency
- initial_balance
- balance_date
- sort_order
- status
- notes
- created_at
- updated_at

### 4. 类型字段的中文表现

数据库内部允许使用稳定代码值。

但用户界面必须只展示中文。

例如：

- account_type 对用户显示：基本户、一般户、临时户、现金账户、其他账户
- instrument_type 对用户显示：银行存款、现金、银行承兑、商业承兑、信用证、其他
- input_method 对用户显示：网银导入、手动填写
- status 对用户显示：启用、已注销、停用

## 六、账户初始化的硬约束

创建账户时，期初余额与余额日期必须是正式字段，不能放到备注里糊弄过去。

用户第一次使用系统时，很多账户早就有余额。

所以新增账户弹窗或页面里，必须明确出现：

- 期初余额
- 余额日期

这两个值将作为后续日报承接与滚动计算的起点。

若用户不填，系统必须明确提醒，不允许静默写成零。

## 七、页面与交互要求

### 1. 页面入口

左侧导航必须存在“账户管理”或“主数据中心”入口。

第一阶段建议名称用“账户管理”，更贴近财务直觉。

### 2. 页面布局

页面分为左右两部分更稳。

左边树形区展示：大区或板块 → 法人单位 → 账户。

右边详情区展示当前选中对象的详细信息和编辑表单。

### 3. 列表展示要求

树或表中至少能直观看到：

- 所属大区或板块
- 法人单位简称
- 开户银行
- 开户行全称
- 账户简称
- 账号尾号或脱敏账号
- 账户类型
- 录入方式
- 当前状态

### 4. 新增账户表单要求

新增账户时表单字段建议按这个顺序：

1. 所属大区或板块，可空
2. 所属法人单位
3. 开户银行
4. 开户行全称
5. 银行账号
6. 账户简称
7. 账户类型
8. 资金工具类型
9. 录入方式
10. 币种
11. 期初余额
12. 余额日期
13. 备注

### 5. 停用要求

账户、法人单位、大区或板块都不允许真删。

只允许停用或标记已注销。

因为后面历史日报、历史交易、导出报表都可能还要引用旧对象。

## 八、API 设计要求

这一块至少要完成下面这些接口。

### 大区或板块

- 创建大区或板块
- 查询列表
- 更新
- 停用
- 排序调整，可后置但要预留字段

### 法人单位

- 创建法人单位
- 查询列表
- 根据大区筛选
- 更新
- 停用

### 账户

- 创建账户
- 查询账户列表
- 按法人筛选
- 按状态筛选
- 更新账户
- 停用账户
- 读取单个账户详情

## 九、数据库和服务层要求

### 数据库层

在 `backend/db/tables.py` 中建立对应表定义。

在 `backend/database.py` 中完成连接和初始化。

### 服务层

本块业务逻辑必须主要放在：

- `backend/services/account_service.py`

若还需要法人或大区服务，可拆成：

- `backend/services/entity_service.py`
- `backend/services/division_service.py`

路由层只负责接收参数、调用服务、返回结果。

不要把校验逻辑和数据库操作散落在路由里。

## 十、前端实现要求

前端建议先完成以下页面与组件。

- `frontend/src/views/AccountManage.vue`
- `frontend/src/api/index.js` 中补账户管理相关请求
- 可复用表单组件或抽屉组件
- 树形或左侧层级组件

使用 Ant Design Vue 的表单、表格、树形控件、抽屉或弹窗即可。

第一阶段优先保证清楚、稳定、中文，不追求炫技动画。

## 十一、自测场景

Cloud Code 完成这一块后，必须至少用下面这些场景自测。

### 场景 1

创建一个不带大区的小公司。

创建一个法人。

创建一个中行基本户。

录入期初余额 200000，余额日期 2026-04-01。

保存成功后，列表中可见。

### 场景 2

创建“华北大区”。

创建“某某建设集团有限公司”。

在该法人下创建两个账户：

- 中行基本户，网银导入
- 现金账户，手动填写

保存后树形结构正确展示。

### 场景 3

将一个旧账户标记为已注销。

它不再出现在默认启用列表里，但历史仍可查询。

### 场景 4

尝试创建一个不填期初余额的账户。

系统必须阻止并提示。

## 十二、本块完成标准

只有同时满足下面这些条件，这一块才算完成。

用户能在中文界面中维护大区、法人、账户。

账户创建时必须填写期初余额和余额日期。

所有主数据可编辑、可停用。

前后端接口打通。

数据库表结构稳定可用。

列表展示能让财务一眼识别“哪个公司、哪个账户、是什么类型、当前状态如何”。

## 十三、本块完成后才能进入下一块

只有本块完成后，才允许进入 AI 配置与 Agent 骨架，或者进入银行流水导入。

因为没有主数据，后面所有导入、识别、日报生成都会失去落点。
