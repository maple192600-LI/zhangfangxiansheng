# 业务规则总账

> 记录系统中所有硬性业务约束。这些规则是代码中实际执行的，不是建议。

---

## 一、流水数据规则

### R1. 收入支出互斥
一条流水记录的收入金额和支出金额不能同时大于 0。
- 代码位置：FundEvent 表的 `ck_fund_events_amount_mutex` 约束
- 正常：收入 10000、支出 0，或收入 0、支出 5000
- 异常：收入 10000、支出 5000

### R2. 金额非负
收入金额和支出金额都必须 ≥ 0。
- 代码位置：FundEvent 表的 `ck_fund_events_amount_nonneg` 约束

### R3. 流水状态枚举
流水状态只有四种值：正常、待确认、异常、已作废。
- 代码位置：FundEvent 表的 `ck_fund_events_state_enum` 约束
- 正常 → 参与所有报表计算
- 待确认/异常 → 不参与报表计算，出现在异常中心
- 已作废 → 不参与任何计算，保留记录

### R4. 流水来源枚举
来源只有五种值：网银导入、手工录入、现金录入、票据录入、财务公司单据。
- 代码位置：FundEvent 表的 `ck_fund_events_source_enum` 约束

### R5. 滚动余额
rolling_balance 字段记录该账户在该日期的累计余额。余额重建功能（基础数据表的"重建余额"按钮）会重新计算所有账户的滚动余额。

## 二、实体/账户匹配规则

### R6. 法人匹配优先级
手工录入时，输入的法人简称/编码按以下优先级匹配：
1. entity_code 精确匹配
2. short_name 精确匹配
3. 账户别名间接匹配
4. name 包含匹配（只匹配到 1 个时才命中）

匹配失败 → 流水标记为"待确认"。

### R7. 账户匹配优先级
手工录入时，输入的账户编号/名称按以下优先级匹配：
1. account_code 精确匹配
2. account_aliases 表的 alias_text 精确匹配
3. account_alias 精确匹配

匹配失败 → 流水标记为"待确认"。

## 三、报表计算规则

### R8. 只统计"正常"状态
所有报表查询只统计 `state = '正常'` 的流水。待确认、异常、已作废的记录不参与计算。

### R9. 期初余额计算
期初余额 = 账户的 initial_balance + 从 balance_date 到查询开始日期前一天的净变动。
- 代码位置：`base_data_service.get_opening_balance`

### R10. 日报汇总
资金日报按法人汇总其名下所有 `include_in_daily_report = true` 的账户。
- 每个法人一行：期初 = 该法人所有账户期初之和
- 收入 = 该法人所有账户收入之和
- 支出、期末同理

### R11. 余额表小计
账户余额表按法人分组，每个法人的账户列完后插入一行小计。小计行的金额为该法人所有账户的合计。

### R12. 日记账按日聚合
现金日记账按账户分块，块内按日聚合。同一天的收入/支出合并为一条。

## 四、导入批次规则

### R13. 每次导入创建批次
每次上传文件或快速录入保存都会创建一个 ImportBatch 记录，用于追溯数据来源。

### R14. 确认提交前清除旧数据
银行导入和手工 Excel 提交时，会先清除该批次已有的 FundEvent，再用最新解析结果重建。这确保了重复预览不会产生重复数据。

### R15. 手工快速录入直接写入
快速录入保存时直接写入 FundEvent（不经过预览），跳转到预览页面只是展示已保存的结果。

---

## 五、报表模板规则

### R16. 模板缺失不阻断
报表页面有默认列配置。如果数据库中没有对应报表类型的模板，使用内置默认列显示。模板是增强层，不是前置条件。

### R17. 默认模板唯一
每个报表类型最多只有一个默认模板。设置新默认模板时，旧的默认模板自动取消默认。

---

## 六、导出文件名规则

### R18. 统一中文文件名
所有报表导出的文件名统一使用中文报表名 + 日期区间。详细规则见《报表取数说明》中导出部分。日期为空时有明确的 fallback（"起"/"截至"/"全部"）。

---

## 七、Agent 产物规则

### R19. 解析器状态流转
ParserArtifact 的状态只有三种：draft（草稿）→ active（激活）→ retired（退役）。
- 只有 active 状态的解析器能被银行导入匹配和使用
- 同一 kind + account_code 下，优先使用版本号最高的 active 解析器

### R20. 规则产物状态流转
RuleArtifact 的状态只有三种：draft → active → retired。流转规则同解析器。

### R21. 禁止 LLM 参与
解析器执行（`run_parser`）和规则执行（`run_rule`）过程中禁止调用 LLM。这是确定性执行约束。
- 代码位置：`artifact_runtime.py` 的 Runtime constraints 注释

### R22. AST 白名单
解析器和规则中的代码只能导入白名单内的模块（如 openpyxl、datetime 等）。禁止导入 os、subprocess、网络请求等模块。
