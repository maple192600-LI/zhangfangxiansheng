## 3. 银行流水 AI 映射（🟡唯一真正在跑的 LLM 功能）的 4 个缺口

| 缺口 | 严重度 | 证据 | 修复工作量 |
|---|---|---|---|
| A. prompt 只拼了表头，**样本数据压根没传进去**（前端 `BankImport.vue:447` 不传 `sample_rows`，后端 `bank_import_service.py:392-398` 参数通道存在但空值）→ AI 识别精度打折 | 🔴 高 | 前端调用漏参数 | 0.5 人日 |
| B. **样本不脱敏，直接明文给 LLM**（`bank_import_service.py:394-398`）→ 隐私风险 | 🔴 高 | LLM 调用前无脱敏 | 1 人日（做 `standard` + `strict` 两档） |
| C. JSON 解析失败只抛错，无降级（`bank_import_service.py:443`）→ 模型输出轻微偏差整个流程失败 | 🟡 中 | 无正则 fallback、无重试 | 0.5 人日 |
| D. `agent_configs.ai_config_id` 字段定义了"parser_assistant 用哪个 provider"，但业务流程**根本不读它**，永远用全局 default provider → 两套配置体系未打通 | 🟡 中 | `bank_import_service.py:375-377` 硬编码取 default | 0.5 人日 |

**小计**：修好这 4 个缺口 = **2.5 人日**，银行流水 AI 映射就从 🟡 升到 🟢。

## 5. 三条路线的真实对比

### 路线 A：**继续 v3 重建**（当前在走的路）

| 维度 | 值 |
|---|---|
| 剩余工作量 | ~33 人日（P0-T4 ~ P2-T4） |
| 何时用户感知到改善 | P1-T1 完成后（约 8-10 人日） |
| v2 已有的 20 条配置 | 需写迁移脚本（非自动） |
| 产出 | 工业级 AI 安全架构：沙箱 + artifact 版本化 + 审批链 + 5 skill Agent |
| 风险 | 33 人日期间 v2 不再动，任何 v2 bug 也先放着 |
| 适合谁 | 这是**开源/上市场卖的产品**，要"合规审计"为生的场景 |

### 路线 B：**停 v3，在 v2 上补缺口**

| 维度 | 值 |
|---|---|
| 剩余工作量 | ~12-15 人日（见 §6 优先级） |
| 何时用户感知到改善 | 2-3 天就能看到第一版 |
| v2 已有的 20 条配置 | 继续用，不动 |
| 产出 | AI 功能覆盖 80% 用户场景，但无 artifact / 无沙箱 |
| 风险 | AI 产出代码无审计链，企业客户可能挑刺 |
| 适合谁 | **个人 / 小团队内部用**，重用户体验不重合规 |

### 路线 C（推荐）：**混合·保留基建·优先补 v2 AI 业务**

| 维度 | 值 |
|---|---|
| 剩余工作量 | 短期 ~12 人日（v2 AI 补齐），长期 v3 基建不急 |
| 当下保留 | P0-T1/T2/T3 已做的 guards + v3 表 + 37 个基元库（**是 asset，不拆**） |
| 暂缓 | P0-T4/T5（沙箱、Agent Harness）→ 等 v2 业务稳定后再搞 |
| v2 已有的 20 条配置 | 继续用 |
| 新加功能的纪律 | 能用基元库的就用（鼓励但不强求） |
| `fund_events` 表 | 先不强切，新写的功能可选择往这里写 |
| 何时用户感知到改善 | 3 天内 |

## 8. 我推荐的决策

**路线 C：混合 · 保留 v3 基建、优先补 v2 AI 业务。**

理由：
1. v2 已经在用户手上跑，有 20 条真实配置，不能简单丢
2. 用户今天要的是"完善 AI 功能"，不是"拿着安全审计报告参加招投标"
3. P0-T1/T2/T3 已经做完的 guards + v3 表 + 37 基元库是**纯增量**，不拆它们一点代价没有
4. v3 基建（P0-T4/T5）等到 v2 业务稳定后、或者用户要对外交付时，再启动还来得及
5. 13.5 人日在 v2 上的投入，**能让你看得见的 AI 功能从"只有一个映射建议"提升到"银行/手工/报表三个场景都有 AI 辅助 + 隐私脱敏 + 逐列编辑"**
