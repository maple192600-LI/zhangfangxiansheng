# 01 · 范围与执行顺序

本文档定义当前整理阶段的产品范围、已知缺口和执行顺序。

## 1. 产品边界

当前产品聚焦本地财务工作台：

- 主数据中心。
- 银行流水导入。
- 手工流水录入和上传。
- 基础数据表。
- 报表生成与导出。
- 规则中心。
- Agent 规则创建、规则维护、Skill、Memory。
- 备份、日志、基础认证和本地启动。

当前不做：

- 发票 OCR 正式流程。
- 费用审批、合同审批。
- 多人协作和集中部署。
- 更换主要技术栈。
- 让用户直接编辑 JSON、正则、SQL 或字段映射。
- 在确定性执行阶段调用 LLM。

## 2. 目标主链路

```text
主数据初始化
  -> 上传银行流水、手工流水或报表模板
  -> 匹配 Parser 或 Rule
  -> 无规则时由 Agent 创建规则
  -> 用户审核规则和样本结果
  -> 规则中心保存
  -> 确定性执行
  -> 预览
  -> 用户确认
  -> fund_events
  -> 报表生成与导出
```

## 3. 当前高风险缺口

| 缺口 | 影响 | 判断 |
| --- | --- | --- |
| 银行导入仍存在旧模板映射路径 | 会绕开 Parser artifact 和规则中心 | 必须迁移 |
| Artifact runtime 不执行 Parser/Rule | 已审核规则无法成为日常执行资产 | 必须修复 |
| Parser/Rule 创建结果不稳定 | Agent 只生成占位草稿时无法上线 | 必须修复 |
| `fund_events` 没有形成全链路事实源 | 报表和基础数据表无法可信 | 必须修复 |
| API 契约与真实代码数量不一致 | guard 红灯，接口入口混乱 | 先盘点再合并 |
| 旧表名和旧 API 命名残留 | 与稳定概念冲突 | 需要迁移计划 |
| 文档中仍有旧阶段和旧方案 | AI coding 会继续照旧错路线实现 | 已开始治理 |

## 4. 执行顺序

1. 完成文档权威审计，清楚标注 active、frozen、needs-rewrite、reference。
2. 修正旧执行文档，不再让 AI 读取旧映射路径作为当前目标。
3. 盘点代码中的旧新并存点，包括 API、服务、页面、数据库表、runtime。
4. 修复银行 Parser 主链路。
5. 修复手工 Parser 主链路。
6. 修复报表 Rule 主链路。
7. 统一规则中心展示和审核记录。
8. 统一 Skill 与 Memory 的创建、查看、脱敏、删除。
9. 修复 guard 红灯。
10. 跑测试、guard、前端构建和浏览器主链路验收。

## 5. GitHub 开源准备顺序

1. 保持本地 `main` 可运行。
2. 清理运行时数据、依赖目录、日志、上传文件和本地密钥。
3. 建立公开 GitHub 仓库。
4. 关联远程仓库：`git remote add origin <repo-url>`。
5. 推送 `main` 到远程仓库。
6. 在 GitHub 保护 `main`，要求 PR、检查通过后才能合并。
7. 后续所有功能、修复、文档、迁移都从短分支开始。
8. 每个分支开 PR，PR 通过测试、guard、review 后合并。
9. 发布稳定版本时打 tag，并在 release notes 中说明功能、修复、迁移和已知限制。

## 6. 阶段验收

### 文档整理验收

- 根 `AGENTS.md`、`CLAUDE.md`、`docs/README.md` 指向一致。
- 旧文档都有状态，不再默认可信。
- `new/` 明确为参考区。
- 非冻结权威文档不再指导旧映射路径。

### 银行导入验收

- 首次样本能创建 Parser。
- Parser 经用户审核后进入规则中心。
- 同银行同格式不同账户可复用同一 Parser。
- 确定性执行后进入预览，确认后写入 `fund_events`。

### 报表验收

- 报表只从 `fund_events` 和 Rule 取数。
- 模板首次上传可创建 Rule。
- 已审核 Rule 可重复生成报表。

### Agent 验收

- Skill 使用 `SKILL.md`。
- Memory 分类型保存。
- 敏感明细不隐藏写入 Memory。
- 用户可查看、脱敏、删除记忆。

### 工程验收

- 后端测试通过。
- 前端构建通过。
- guard 给出明确结果。
- 浏览器跑通主链路。
