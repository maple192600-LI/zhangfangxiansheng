# 08 · 防跑偏机制

本项目防跑偏的核心目标是：AI 不再把旧文档、旧入口、旧代码和平行实现继续扩散。

## 常见跑偏模式

| 模式 | 表现 | 处理 |
| --- | --- | --- |
| 权威漂移 | 读取失效文档或第三方参考项目当作需求 | 回到 `AGENTS.md` 和 `docs/README.md` |
| 命名分裂 | 新增版本后缀命名或平行 Agent 实现 | 停止扩散，进入迁移清单 |
| 数据流断裂 | 页面按钮各自调用不同旧接口 | 收束到上传、规则、预览、确认、入库主链路 |
| 运行时越权 | 导入或报表生成阶段调用 LLM | 回滚该路径，改为 Parser/Rule 确定性执行 |
| 用户被迫配置 | 暴露 JSON、正则、SQL、字段映射 | 改为 Agent 创建规则、用户审核结果 |
| 守卫绕过 | 为了通过测试创建替代文件 | 删除替代路径，修真实缺陷 |
| Git 流程绕过 | 直接改 `main` 或无 PR 合并 | 回到短分支、PR、检查、review 流程 |
| 局部替代全局 | 只处理单一问题，忽略其他整改类别 | 回到迁移清单，标注本 PR 覆盖和未覆盖范围 |
| 空入口假完成 | Placeholder、空分类、不可用按钮仍在导航里 | 隐藏或删除入口，只保留能跑通真实工作流的页面 |
| 记忆不可审计 | 隐藏写入敏感流水明细或无法查看、脱敏、删除 | Memory 必须分级、可审计、可删除 |

## 防线

1. `AGENTS.md`：项目级入口和硬规则。
2. `docs/README.md`：文档权威顺序。
3. 冻结契约：schema、字段、白名单、Parser/Rule 边界。
4. 守卫脚本：阻止契约、命名、运行时、目录边界漂移。
5. 测试与浏览器验收：证明主链路真的可用。
6. GitHub Flow：短分支、PR、检查、review、合并。
7. 交付说明：每次说明改了什么、验证了什么、还剩什么。

## Kickoff 检查

每次开工前确认：

```text
1. 当前任务属于文档治理、Agent 主链路、数据流、前端体验、测试守卫中的哪一类？
2. 是否会触碰冻结契约？
3. 是否会新增或删除 API、表、Parser、Rule 或页面入口？
4. 是否可能造成平行实现？
5. 本次完成后要跑哪些测试、守卫和浏览器验证？
6. 本次工作应该在哪个分支完成，PR 描述需要哪些验证证据？
7. 本次覆盖迁移清单中的哪些类别？哪些明确不覆盖？
8. 本次是否改动前端入口？保留、合并、隐藏、删除的理由是什么？
```

无法回答时，先审计，不写代码。

## 守卫脚本

现有守卫：

- `check_contract_hash.py`
- `check_canonical_schema.py`
- `check_primitives_whitelist.py`
- `check_placeholder_binding.py`
- `check_api_inventory.py`

新增守卫：

- `check_no_runtime_llm.py`

- `check_product_purity.py`

冻结契约哈希失败时，不允许 AI 自行更新 `contracts.lock`。必须报告用户确认。

## PR 防跑偏要求

PR 是开源仓库的合并入口。每个 PR 必须回答：

- 这个 PR 解决哪个 issue、任务或迁移清单项？
- 是否触碰冻结契约？
- 是否新增 API、数据表、页面或规则资产？
- 是否删除或迁移旧入口？
- 跑了哪些测试、构建、guard 和浏览器验证？
- 是否有用户数据迁移风险？
- 是否覆盖迁移清单中的具体类别或编号？
- 如果涉及前端，是否给出全量点击路径审计和导航取舍？

不能回答这些问题时，不应合并。

## 交付要求

每个可交付任务都要说明：

- 文件清单。
- 自动化验证结果。
- 浏览器验证结果。
- 守卫结果。
- 仍未完成或被冻结契约阻塞的事项。
