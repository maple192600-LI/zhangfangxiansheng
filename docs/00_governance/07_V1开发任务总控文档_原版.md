# V1 开发任务总控文档

## 一、文档目的

本文档用于指导 Cloud Code 按固定顺序完成“账房先生”第一阶段开发。

这份文档只服务一件事。

让 Cloud Code 在有限上下文里，清楚知道当前阶段要做什么、先做什么、后做什么、每一步做到什么程度算完成、哪些内容当前不能碰。

本文件的执行优先级低于《需求总览》《目标用户与使用习惯约束》《技术选型约束》，高于临时发挥。

## 二、V1 的唯一目标

V1 只完成“出纳资金日报”第一阶段最小可用闭环。

完成后，用户应能在 Windows 电脑上双击启动程序，通过浏览器进入本地系统，完成以下动作。

创建大区或板块。

创建法人单位。

创建银行账户并录入期初余额与余额日期。

导入多家银行的流水文件。

对新银行模板进行首次识别并保存模板。

对无法导流水的账户做手工表格式录入。

按日期区间生成资金日报。

查看、筛选、搜索、导出、打印资金日报。

查看基础数据看板。

配置 AI 厂商接口。

落下 Agent 基础骨架、总管 Agent、解析助手和 shared 公共区。

记录日志，支持基础备份恢复与导入批次回滚。

## 三、V1 明确不做的内容

以下内容只做预埋、占位、目录和接口准备，不进入当前阶段深做。

发票 OCR 识别正式流程。

费用审批。

合同管理。

用户自建 Agent。

多 Agent 协作。

多人协同使用。

集中式数据库部署。

桌面壳封装。

复杂 BI 仪表盘。

任何超出 V1 主链路的大块未来功能。

## 四、Cloud Code 开工前必须先读的文件

每次开始 V1 开发前，Cloud Code 必须先完整阅读以下文件。

1. `zhangfang-需求总览.md`
2. `03_目标用户与使用习惯约束.md`
3. `04_技术选型约束.md`
4. `资金日报系统_需求文档_v3_按用户修订版.md`
5. 本文件 `05_V1开发任务总控文档.md`
6. 根目录 `CLAUDE.md`

若上述文件之间出现冲突，以约束文件和当前修订后的主需求为准，并先提出冲突点，再执行。

## 五、V1 的完成标准

只有同时满足下面这些条件，V1 才算完成。

应用可在 Windows 上本地运行。

用户双击后可自动进入浏览器界面，无需命令行。

界面全中文。

账户支持期初余额初始化。

日报支持区间生成。

日报支持导入流水与手工录入两种来源。

日报支持筛选、搜索、导出和打印。

新银行模板可完成首次识别并保存模板配置。

至少存在总管 Agent 与解析助手两类内置 Agent。

数据可备份，可恢复，可做导入批次回滚。

主流程在真实样本上可跑通。

## 六、项目目录硬约束

V1 开发时，项目目录必须朝下面这个方向建立，不允许随意改成别的结构。

```text
zhangfang/
├── CLAUDE.md
├── zhangfang-需求总览.md
├── 03_目标用户与使用习惯约束.md
├── 04_技术选型约束.md
├── 05_V1开发任务总控文档.md
│
├── backend/
│   ├── main.py
│   ├── config.py
│   ├── database.py
│   ├── requirements.txt
│   ├── core/
│   ├── db/
│   ├── api/
│   ├── services/
│   ├── agents/
│   ├── parsers/
│   └── data/
│
├── frontend/
│   ├── package.json
│   ├── vite.config.js
│   └── src/
│
└── build/
```

允许在不破坏主结构的前提下补充子目录。

不允许在 V1 里引入第二套前端框架。

不允许跳过 `backend/services` 直接把复杂业务逻辑全塞进路由。

不允许把 Agent、解析器、数据库访问、导出逻辑写成一坨耦合代码。

## 七、V1 开发顺序总览

V1 必须按下面顺序推进。

第 0 步，搭脚手架与运行骨架。

第 1 步，建数据库与基础表。

第 2 步，建主数据管理闭环。

第 3 步，建 AI 配置闭环。

第 4 步，建解析器模板骨架。

第 5 步，建银行流水导入预览链路。

第 6 步，建手工表格式录入链路。

第 7 步，建区间日报生成链路。

第 8 步，建日报查看、筛选、导出、打印。

第 9 步，建基础看板。

第 10 步，建 Agent 基础骨架。

第 11 步，建日志、备份、回滚。

第 12 步，联调、测试、修正。

任何步骤未通过成功标准，不允许进入下一步。

## 八、第 0 步：脚手架与运行骨架

### 要做什么

建立前后端基础工程。

后端使用 FastAPI。

前端使用 Vue 3 + Vite + Ant Design Vue。

后端负责挂载前端静态资源。

启动后自动打开浏览器本地地址。

### 需要落的文件

`backend/main.py`

`backend/config.py`

`backend/requirements.txt`

`frontend/package.json`

`frontend/src/main.js`

`frontend/src/App.vue`

### 本步成功标准

本地执行开发启动命令后，浏览器可打开首页。

首页至少能显示应用名、左侧菜单占位、主内容区占位。

前后端基础通信可通。

## 九、第 1 步：数据库与基础表

### 要做什么

完成 SQLAlchemy 连接、SQLite 初始化、表创建、基础迁移能力。

优先建立以下表。

`divisions`

`entities`

`accounts`

`fund_events`

`parser_templates`

`custom_fields`

`custom_field_values`

`ai_configs`

`daily_reports`

`system_config`

`operation_logs`

`agent_configs`

### 必须注意

账户表必须有期初余额和余额日期。

账户表必须有开户银行、开户行全称、账户简称、账户类型、工具类型、录入方式。

交易表必须预留部门、费用类型等后续字段。

### 本步成功标准

数据库可自动初始化。

所有核心表都可成功创建。

至少完成一次插入和查询验证。

## 十、第 2 步：主数据管理闭环

### 要做什么

完成大区或板块、法人单位、银行账户管理页面与接口。

账户创建时必须支持录入期初余额和余额日期。

账户支持启用、停用、注销状态。

账户支持网银导入和手动填写两种录入方式。

### 页面建议

`AccountManage.vue`

支持树形结构查看：大区或板块 → 法人单位 → 银行账户。

### 本步成功标准

用户可完整创建一套主数据。

至少能创建 1 个大区、2 个法人、3 个账户。

账户列表能清晰显示中文字段。

## 十一、第 3 步：AI 配置闭环

### 要做什么

完成 AI 厂商配置页。

支持至少以下配置方式。

智谱。

Kimi。

通义。

OpenAI 兼容接口。

Ollama。

用户只需要粘贴 Key 和必要地址即可。

必须支持测试连接。

### 本步成功标准

至少 1 个厂商可保存成功并测试通过。

配置数据加密存储。

页面全中文。

## 十二、第 4 步：解析器模板骨架

### 要做什么

建立银行流水模板识别和模板存储骨架。

当前阶段只要求做到。

能保存模板元信息。

能保存列映射关系。

能保存样本表头。

能记录模板由 AI 创建还是人工修订。

### 必须注意

当前阶段不要求把模板系统做成高度智能。

先把“首次识别后能保存，下次能复用”跑通。

### 本步成功标准

系统可保存至少 1 份银行模板。

再次导入同类文件时，能命中同一模板。

## 十三、第 5 步：银行流水导入预览链路

### 要做什么

完成文件上传、结构化读取、模板匹配、预览展示、确认写入。

优先支持 Excel、CSV。

对银行流水优先走结构化读取，不走 OCR。

### 用户流程

选择日期范围。

上传一批文件。

系统读取表头并尝试匹配模板。

匹配成功则直接预览。

匹配失败则进入“新模板识别”流程。

用户确认后写入数据库。

### 本步成功标准

至少用 2 家不同格式银行样本成功导入。

预览表里能看到日期、摘要、对方名称、收入、支出、余额、所属账户等核心字段。

确认写入后，数据库出现对应交易记录。

## 十四、第 6 步：手工表格式录入链路

### 要做什么

完成无法导流水账户的手工表格式录入页面。

页面必须尽量接近 Excel 录入感。

以横向表格形式录入。

允许连续录入多行。

自动计算余额。

支持自定义字段列展示。

### 必须注意

不要做成“再录一笔”的繁琐按钮流程。

要让用户像填表一样直接往下填。

### 本步成功标准

用户可选定某个账户后，连续录入多笔交易。

提交后，数据库正确写入。

余额按日期顺序计算正确。

## 十五、第 7 步：区间日报生成链路

### 要做什么

实现资金日报核心生成逻辑。

用户可选择起始日期和结束日期生成日报。

系统按起始日之前最后一个有效余额承接期初。

系统抓取区间内全部导入与手工交易。

按真实日期顺序生成平铺日报数据。

### 必须注意

日报是输出视图。

底层真相来源是数据库中的交易与主数据。

日报标题要支持区间，如“2026年4月1日到4月3日资金日报”。

### 本步成功标准

至少完成 1 天日报与 3 天区间日报两类测试。

日报内每笔交易保留真实日期。

区间期初、区间合计、区间期末计算正确。

## 十六、第 8 步：日报查看、筛选、导出、打印

### 要做什么

完成日报页面正式视图。

采用平铺大表形式。

支持按日、周、月、年、自定义区间筛选。

支持按大区、法人、账户筛选。

支持搜索、排序、导出 Excel、打印。

### 页面要求

主表格必须清楚显示。

板块或大区。

法人单位。

账户。

日期。

期初余额。

摘要。

对方名称。

借方。

贷方。

余额。

用户自定义列。

### 本步成功标准

日报页面可在真实样本数据下正常查看。

导出 Excel 不丢字段。

打印视图可用。

## 十七、第 9 步：基础看板

### 要做什么

完成最基础的数据看板页面。

先只做轻量能力。

今日总收入。

今日总支出。

账户总余额。

近 7 天或近 30 天收支趋势。

账户余额占比。

### 必须注意

看板是辅助视图。

不能抢主表格主战场。

### 本步成功标准

看板可基于当前数据库真实数据生成。

支持按日、周、月、年切换。

## 十八、第 10 步：Agent 基础骨架

### 要做什么

完成 Agent 工作区隔离骨架。

建立 `agents/shared/` 公共区。

建立 `agents/master/` 总管 Agent。

建立 `agents/parser-assistant/` 解析助手。

每个 Agent 目录必须预留以下文件。

`SOUL.md`

`AGENT.md`

`USER.md`

`MEMORY.md`

`TOOLS.md`

`RULES.md`

`WORKFLOW.md`

### 当前阶段只要求

能加载 Agent 基础配置。

能在界面里看到内置 Agent。

能完成最基础的对话入口或调用入口。

### 本步成功标准

总管 Agent 和解析助手目录结构完整。

shared 公共区存在。

Agent 引擎可以读取对应配置文件。

## 十九、第 11 步：日志、备份、回滚

### 要做什么

完成操作日志记录。

完成数据库备份恢复。

完成按导入批次回滚。

### 必须注意

日志至少记录导入、手工录入、修改、删除、导出等动作。

备份恢复要做到普通财务人员能通过界面完成。

### 本步成功标准

用户可手动执行一次备份。

用户可恢复一次备份。

用户可回滚某个导入批次。

## 二十、第 12 步：联调与验收

### 要做什么

用真实或脱敏样本做端到端联调。

至少覆盖以下场景。

新建主数据。

录入期初余额。

导入两家银行流水。

新模板首次识别。

手工补录一组数据。

生成单日和区间日报。

筛选和导出。

查看看板。

AI 配置保存。

### 本步成功标准

所有主链路动作都可完成。

不存在明显阻塞性报错。

日报金额与期末余额可人工核对通过。

## 二十一、Cloud Code 每一步输出格式要求

Cloud Code 在完成每一步时，必须输出四类信息。

本步新增或修改了哪些文件。

本步完成了哪些功能。

本步还有哪些已知风险。

本步为什么已经达到进入下一步的条件。

如未达到条件，必须停在当前步，不能自行跳步。

## 二十二、Cloud Code 禁止事项

V1 开发时，Cloud Code 禁止做下面这些事。

自行改技术栈。

自行换数据库。

自行加收费组件。

自行跨阶段做发票、费用、合同主流程。

为了桌面感引入 Electron 或 Tauri。

为了表格体验引入许可证复杂的商业表格依赖。

把核心账务逻辑丢给大模型运行期判断。

跳过测试直接宣称完成。

## 二十三、V1 交付物清单

V1 完成时，至少应交付以下内容。

可运行代码。

可打包版本。

数据库初始化脚本。

主数据管理页面。

AI 配置页面。

导入流水页面。

手工录入页面。

资金日报页面。

基础看板页面。

Agent 基础骨架。

日志、备份、回滚能力。

基础测试说明。

## 二十四、结论

V1 的任务很明确。

先把“出纳资金日报”做成一个普通财务人员真的能打开、真的能录、真的能导、真的能查、真的能补、真的能跑通的本地工具。

只要这个闭环站稳，V2 以后所有能力才有真实根基。

在 V1 未稳定前，任何未来想象都不能抢占主链路。
