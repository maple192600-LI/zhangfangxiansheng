# 技术选型约束

## 一、文档目的

本文档用于固定“账房先生”项目在当前阶段的技术选型边界、允许采用的实现方式、暂不采用的方向、替换条件与升级规则。

它不是技术科普，也不是框架评测文章。

它的唯一作用，是让 Claude Code 与后续任何协作开发者在实现项目时，不要因为一时的技术偏好、流行趋势或想当然，偏离本项目当前最合适的路径。

从本文件生效开始，技术选型必须服务业务落地、用户门槛、成本控制、可维护性与未来扩展，不允许脱离项目现实独立发挥。

## 二、选型总原则

本项目的技术选型，必须同时满足六个原则。

第一，先保证能落地，再追求漂亮架构。

第二，先保证用户能用，再追求技术先进。

第三，先保证开发与维护可控，再追求理论最优。

第四，先保证第一阶段能跑通，再谈远期扩展。

第五，先保证本地部署、低门槛、低成本，再谈复杂基础设施。

第六，先保证规则可追溯、结果可复查、数据可留痕，再谈 AI 自动化程度。

## 三、当前阶段技术底座

当前阶段的基础技术路径固定如下。

后端采用 Python 3.11+。

后端框架采用 FastAPI。

前端采用 Vue 3 + Vite。

UI 组件采用 Ant Design Vue。

图表采用 Apache ECharts。

数据库第一阶段采用 SQLite，并通过 SQLAlchemy ORM 隔离数据库实现细节。

数据处理采用 Polars + openpyxl。

AI 厂商接入采用统一 Provider 层，支持多家模型厂商与 Ollama 本地模型。

OCR 集成采用 PaddleOCR，并直接放在后端能力层内。

打包方式采用前端编译静态资源 + 后端挂载 + PyInstaller 打包为本地可执行程序。

这套路径已经与你的总览文档、需求文档和当前产品边界对齐，后续开发必须在这套底座上推进，除非有明确重审理由。

## 四、为什么当前阶段必须采用这套底座

原因并不复杂。

这个项目的第一阶段核心问题，是多法人、多账户、多来源的资金日报处理。

它的真正技术难点在于文件读取、模板识别、标准化解析、规则引擎、余额承接、异常处理、导出打印、历史补做、手工补录与追溯留痕。

这些问题的主战场在后端数据处理，不在桌面壳，不在低层系统编译能力，也不在炫技型前端工程。

Python 在文件处理、表格处理、OCR 集成、规则计算、脚本可审计性与 AI 生态接入上，都更适合当前阶段。

Vue 3 + Ant Design Vue 更适合承载“现代化但熟悉的企业后台界面”，能兼顾好看、免费、开源、中文场景与 Cloud Code 的实现稳定性。

SQLite 适合第一阶段单机本地部署，零配置、便于备份、便于打包、便于普通用户使用；SQLAlchemy 的存在，又保证未来切换 PostgreSQL 时不需要大范围重写业务层。

因此，这不是妥协方案。这是当前阶段最稳的方案。

## 五、前端技术约束

前端层必须遵守下面这些规则。

第一，前端以 Vue 3 为唯一主框架，不引入第二套前端框架并存。

第二，组件体系以 Ant Design Vue 为主，不允许为了单个视觉效果再引入另一整套大型 UI 框架。

第三，图表体系以 ECharts 为主，不允许同一阶段同时混用多套图表库。

第四，页面必须围绕表格主战场展开，图表、卡片、提示条都只能作为辅助视图。

第五，视觉样式必须遵守现有主题皮肤规范，前端开发时应把业务结构层、页面布局层、通用组件层、主题样式层、微文案层拆开，避免视觉修改牵动业务结构。

第六，前端必须全中文，用户看见的账户类型、状态、按钮、配置项、AI 厂商、错误提示都必须是中文可理解表达。

## 六、表格能力约束

本项目第一阶段一定要有强表格能力，但第一阶段不追求在浏览器里复刻完整 Excel。

当前阶段优先目标是把下面这些能力做稳定。

大表展示。

筛选。

搜索。

排序。

冻结表头。

日期区间查看。

行级核对。

异常标识。

导出 Excel。

打印。

手工表格录入。

因此，第一阶段允许使用 Ant Design Vue 原生表格能力配合自定义封装完成日报处理台和查询页。

如果第一阶段后期确认大表性能、编辑体验、虚拟滚动体验确实不足，再单独立项评估引入额外开源表格组件。

在此之前，不得为了“像 Excel”三个字，提前引入付费或许可证复杂的重型表格依赖。

## 七、付费依赖约束

第一阶段禁止把核心主流程建立在必须持续付费的第三方组件或在线服务之上。

核心主流程包括。

导入流水。

手工录入。

日报生成。

筛选查询。

导出打印。

异常处理。

主数据维护。

基础看板。

AI 配置。

Agent 基础骨架。

如果某个依赖的免费版存在明显许可证风险、商用限制风险、闭源限制风险，第一阶段默认不采用。

如果某个依赖虽然很好用，但会把项目带入按席位付费、按年订阅、按企业授权的路线，第一阶段默认不采用。

只有一种情况例外。

该依赖并不处在主链路，只承担可选增强能力，并且存在可被替换的基础实现。

## 八、后端技术约束

后端必须坚持“工程为主、AI 为辅”的实现路径。

凡是能通过确定性脚本完成的能力，必须优先用工程方式实现。

包括文件读取、字段映射、日期解析、金额校验、余额滚动、重复检查、摘要规则命中、导出打印、异常留痕等。

不得把这些核心计算逻辑直接外包给模型在运行期自由发挥。

AI 在后端的职责，应限定在以下范围。

首次初始化辅助。

新模板解析配置生成辅助。

规则草案生成辅助。

低频复杂配置辅助。

自然语言查数与分析辅助。

Agent 解释与引导辅助。

运行期的账务归属、正式摘要、余额正确性、去重决策等关键判断，必须由规则、脚本、校验或人工确认兜底。

## 九、OCR 与文件处理约束

OCR 必须作为应用自身的后端能力存在。

它不能依赖 Claude Code 的技能环境，也不能把运行期识别能力建立在外部对话工具的临时技能调用上。

第一阶段 OCR 的定位是后续发票模块的底层能力预埋，不是当前出纳日报模块的主链路。

当前阶段应先把 OCR 集成框架、依赖管理、调用封装、错误处理、批量输入边界、输出结构统一方式设计好，但不要求把发票模块一并做完。

对银行流水类文件，优先走结构化文件读取链路，不走 OCR 链路。

对发票、票据、图片、扫描 PDF 等非结构化文件，后续模块再接 OCR 处理链。

## 十、数据库约束

数据库第一阶段采用 SQLite。

这里必须强调的是，当前采用 SQLite，不代表把未来锁死在 SQLite。

当前采用 SQLite 的原因，是它符合第一阶段单人本地使用、双击即用、零配置、易备份、易打包、易恢复的目标。

同时，业务层通过 SQLAlchemy ORM 实现，这意味着后续要切换 PostgreSQL 时，理论上的主要变更应集中在连接配置、迁移脚本与少量部署方式上，而不是整个项目重写。

因此，数据库策略固定为下面这条。

第一阶段用 SQLite。

数据库访问必须统一走 SQLAlchemy。

严禁在业务代码里到处手写绑死某个数据库方言的原生 SQL。

如果未来进入多人协作、共享部署、集中式数据库阶段，再单独立项评估 PostgreSQL。

在达到该条件之前，不引入独立数据库服务。

## 十一、打包与运行方式约束

本项目默认运行方式固定为本地可执行程序。

用户双击程序。

程序启动本地后端服务。

系统自动打开浏览器进入本地页面。

用户通过浏览器界面操作系统。

这个运行方式在当前阶段优先级高于桌面壳方案。

原因很明确。

它对普通财务用户已经足够像软件。

它对 Cloud Code 开发与后续维护更稳定。

它避免了 Electron、Tauri 这类多运行时组合在当前阶段带来的额外复杂度。

因此，第一阶段不引入 Electron，不引入 Tauri，也不引入任何需要同时维护 Python、Node、Rust 三套运行时的桌面壳方案。

未来若产品进入成熟稳定阶段，并且确有必要追求独立窗口体验，再单独评估桌面壳封装。

当前阶段不做。

## 十二、Agent 架构约束

Agent 不是本项目的装饰功能。

它是未来产品演进的重要骨架。

因此，Agent 体系必须从一开始就按工作区隔离思路设计。

每个 Agent 都应拥有自己的独立人格目录。

目录中应至少预留以下文件层。

SOUL.md

AGENT.md

USER.md

MEMORY.md

TOOLS.md

RULES.md

WORKFLOW.md

同时必须有 shared 共享区，用于保存公司信息、公共知识、数据结构说明等公共内容。

各 Agent 之间默认上下文隔离，私有记忆隔离，权限隔离。

是否允许读取其他模块的数据，应由权限与工具层控制，而不是默认全开。

第一阶段只需落下 Agent 引擎骨架、总管 Agent、解析助手和共享知识区。

用户自建 Agent、Agent 管理界面、跨 Agent 协作，放在后续阶段。

## 十三、模块化约束

项目目录结构必须围绕“核心引擎 + 独立模块 + 共享能力”展开。

核心层负责数据库、配置、解析引擎、导出引擎、安全、日志、备份、Agent 引擎等通用能力。

模块层负责出纳日报、发票识别、费用审批、合同管理等业务模块。

任何新模块都应尽量建立在统一数据底座、统一配置体系、统一权限体系、统一导出能力与统一 Agent 机制之上。

禁止为了赶进度，在某个模块里私自再造一套配置、再造一套数据库访问方式、再造一套导出逻辑。

## 十四、页面与主题分离约束

后续前端实现时，必须把业务结构和主题皮肤拆开。

主题层只负责配色、圆角、阴影、按钮、导航、卡片、提示区与视觉气质。

页面结构层只负责页面布局、字段摆放、交互流程与业务内容区。

这样后续即便你想换风格、微调颜色、替换按钮皮肤，也不应牵动业务逻辑和页面骨架。

这条约束必须严格执行。

## 十五、文档与开发推进约束

Claude Code 后续开发时，不允许把所有内容塞进一个超长总文件里边走边猜。

项目文档应按下面这组方式拆开。

需求总览文件，用于理解全局。

目标用户与使用习惯约束，用于防跑偏。

技术选型约束，用于锁技术边界。

阶段开发计划，用于当前阶段实施。

模块设计文档，用于逐个功能拆解。

页面设计文档，用于前端落地。

规则字典与字段字典，用于中间层固化。

测试与验收文档，用于完成闭环。

第一阶段开发时，Claude Code 必须始终以当前阶段文档为直接执行依据，以总览文档和本文件为上位约束，不得擅自跨阶段实现大量未来功能。

## 十六、第一阶段必须落地的技术范围

第一阶段技术实现范围固定如下。

本地安装运行。

主数据中心。

账户期初余额初始化。

银行流水批量导入。

新模板辅助识别与解析器生成骨架。

手工表格式录入。

日期区间日报生成。

连续账期承接。

日报平铺查看。

筛选查询。

导出打印。

基础看板。

AI 厂商配置。

Agent 引擎骨架。

总管 Agent。

解析助手。

日志、备份、回滚、异常留痕。

除此之外的能力，哪怕已经在总体蓝图里出现，也应作为预埋和占位，不得挤占第一阶段主链路开发资源。

## 十七、明确禁止的技术倾向

当前阶段禁止出现下面这些倾向。

为了追求桌面感，提前引入 Electron 或 Tauri。

为了追求极致性能，提前引入 Rust 业务层开发。

为了追求低代码感，提前把主逻辑搭在第三方在线平台上。

为了追求快，直接把核心账务逻辑交给大模型运行期决定。

为了追求像 Excel，先引入许可证复杂或未来商用风险高的重型表格组件。

为了追求界面炫酷，牺牲主表格的清晰度和核对效率。

为了图省事，把 OCR、解析、Agent、报表、数据库全写成耦合在一起的大杂烩。

## 十八、何时允许重审技术选型

只有出现下面几类情况时，才允许重审本文件中的技术路线。

第一，第一阶段已跑通，并在真实使用中暴露出明确瓶颈。

第二，某个关键开源依赖停止维护、许可证变化、严重影响商用路径。

第三，未来进入多人协作、共享数据库、集中部署阶段。

第四，Agent 平台和 AI 分析能力增长到当前架构确实承压。

第五，已有路线明显无法承接下一阶段业务目标。

在没有触发这些条件前，任何“换框架”“重构为桌面壳”“切数据库”“上新运行时”的提议，都默认不进入执行。

## 十九、对 Claude Code 的直接命令

Claude Code 在后续开发中，必须遵守以下执行规则。

只在当前选定技术栈内做实现。

不自行添加新的核心依赖。

不自行更换数据库方案。

不自行更换前端主框架。

不自行追加收费组件。

不自行跨阶段实现大块未来功能。

若发现当前技术路线无法满足需求，应先输出风险与替代方案，再等待上位文档修订。

## 二十、结论

“账房先生”当前阶段的技术选型目标，不是做出最时髦的系统。

目标是做出最适合当前用户、当前预算、当前开发方式、当前维护能力、当前阶段目标的系统。

能让产品先稳稳跑起来的技术，才是当前最好的技术。

这份文档自发布起，作为项目当前阶段的正式技术边界。
