# 目标用户与使用习惯约束

## 一、文档目的

本文档用于把“账房先生”项目的目标用户、真实使用环境、操作习惯、容错边界、付费心理与接受能力单独固定下来。

这份文档不是市场宣传材料，也不是泛泛而谈的人群画像。它的唯一作用，是约束后续所有需求讨论、页面设计、模块设计、技术选型、交互设计、AI Agent 设计与开发执行。

从现在开始，凡是与本文件冲突的设计想法，都应优先质疑设计本身，而不是怀疑用户。

## 二、目标用户是谁

本项目第一阶段服务的核心人群，是中国企业里的出纳、资金岗、财务人员、财务主管，以及需要查看资金结果和听汇报的管理层。

其中真正高频操作系统的人，主要是出纳和资金岗。他们负责上传银行流水、手工补录、检查异常、重算日报、确认结果、导出打印。

真正频繁提问、看结果、要汇报口径的人，主要是财务主管、老板和管理层。他们未必每天亲自录数据，但会反复追问重点账户余额、今日已付金额、某法人资金情况、某账户是否紧张等问题。

因此，系统既要照顾执行层的录入与核对，也要照顾管理层的查看与汇报。这个双重目标必须同时成立。

## 三、第一阶段用户画像

第一阶段默认用户，不是假想中的数字化成熟企业，也不是懂系统配置的 IT 用户。

第一阶段默认用户应按下面这组画像理解。

他们大多长期使用 Windows 办公电脑。

他们日常最熟悉的工具，是 Excel、金蝶、用友一类财务软件，以及各家银行网银导出的 Excel 或 CSV 文件。

他们对“表格”天然熟悉，对“系统配置”“字段映射”“命令行”“JSON”“端口”“环境变量”天然陌生。

他们对软件的期待非常朴素。

打开就能用。

界面全中文。

不要折腾。

不要多学。

不要多花钱。

出错时能看懂。

找数据时能快速找到。

导出时别变形。

这群人对“高自由度配置”的接受度并不高，但对“像 Excel 一样的操作感”接受度很高。

## 四、他们真实的工作习惯

这群用户的核心工作习惯，可以概括为四个词：看表、筛选、核对、导出。

他们习惯先看到一张完整的大表，再去找某一行、某一列、某一笔问题。

他们习惯通过日期、法人、账户、金额、摘要、对方户名这些字段去筛选、搜索、对账，而不是先理解复杂的信息架构再操作。

他们习惯把“日报”当作当天工作的结果载体，把“导出 Excel”和“打印”当作最终交付动作。

他们对“按列查看”“冻结表头”“快速搜索”“按日周月区间筛选”“复制导出”“平铺核对”这些能力高度敏感。

他们对复杂弹窗链路、隐藏式交互、强引导向导、过度跳转页面的耐受度很低。

## 五、他们面对的真实业务环境

这群用户的工作环境通常很脏、很碎、很不标准。

同一个企业下可能有多个法人主体。

多个法人可能在同一家银行开账户。

同一家银行不同法人导出的模板相似甚至一致。

有的账户可以网银导流水，有的账户不能。

有的资金来自现金，有的来自手工账户，有的来自票据、承兑、信用证、财务公司单据、保证金等特殊载体。

有时候一天几十笔，有时候只有一两笔。

有时候当天做日报，有时候几天后补做。

有时候日报已经做完，又临时插进来一笔付款，要立刻补进去重算。

所以，这个项目面对的是现实财务场景，不是课堂案例，更不是理想化 ERP 流程。

## 六、他们的电脑与技术能力边界

第一阶段默认用户不会使用命令行。

他们通常不知道 CMD、PowerShell、终端、端口、环境变量、依赖安装、数据库服务分别是什么。

他们也不应该被要求理解这些东西。

凡是需要他们手动执行命令、改配置文件、填 JSON、跑 Docker、装 Nginx、手改数据库连接的设计，都应视为违背用户边界。

系统必须以“普通财务人员双击即可运行”为最低使用门槛。

在用户看来，它应该像一个正常安装的软件。

能看到桌面图标。

双击能开。

有中文页面。

能自己配 API Key。

能自己备份恢复。

能自己完成日常工作。

## 七、他们对成本的真实态度

这群用户和你本人一样，对额外成本非常敏感。

他们真正愿意付费的前提，是软件已经能明显替他们省时间、省出错、提升效率。

在此之前，他们不会愿意为了一个表格组件、一个在线服务、一个商业插件、一个高级界面控件再额外买单。

因此，本项目第一阶段的原则应明确写死。

第一，不引入任何必须持续付费才能正常运行的核心能力。

第二，不把基础录入、日报生成、筛选查询、导出打印这些主功能建立在收费 SaaS 或收费组件之上。

第三，允许未来为个别高阶能力预留商业化升级空间，但第一阶段默认必须可在免费或开源基础上完整跑通。

## 八、他们对界面的真实要求

这群用户需要好看，但他们对“好看”的定义，不是炫技。

他们需要的是干净、稳、柔和、现代、有一点温度，同时看数字不累。

他们希望界面比传统财务软件顺眼，也希望比冷冰冰后台更有人味。

可他们绝不会为了视觉效果牺牲清晰度和效率。

所以，界面应满足下面这组要求。

第一，看起来现代，但不互联网营销风。

第二，颜色柔和，适合长期盯表。

第三，主战场始终是表格。

第四，图表和卡片是辅助能力，不可抢掉主表格。

第五，小贴士、空状态、轻提示可以有温度，但不能干扰严肃业务。

第六，异常、归属不确定、金额冲突这类信息必须直白清楚，不能软化。

## 九、他们对 AI 的真实期待

这群用户并不希望 AI 在运行期替他们瞎判断。

他们更希望 AI 帮他们做三类事情。

第一，首次初始化时，帮他们把复杂配置收拢起来。

第二，新银行模板、新资金载体、新规则变化出现时，帮他们快速生成初始规则与解析配置。

第三，在低风险场景里，帮他们归纳下一步、提醒哪里有异常、辅助查数、辅助解释。

他们不能接受的，是 AI 在没有规则、没有校验、没有人工确认的前提下，直接替他们决定某笔钱归哪个法人、摘要怎么写、余额是不是对的。

财务系统中的 AI，必须先守边界，再谈聪明。

## 十、他们希望产品怎么成长

第一阶段，他们接受先做单人本地使用版本。

他们也接受先把出纳资金日报这一块打透，再逐步扩展到发票识别、费用审批、合同管理、AI 分析平台等能力。

他们希望模块是完整预埋、逐步落地。

他们不希望今天先做一个死板的小工具，后面发现什么都接不上。

所以，第一阶段产品必须满足两个方向同时成立。

一方面，当前就能把出纳资金日报这件事做成。

另一方面，底层数据结构和模块边界要为后续发票、费用、合同、Agent、多角色协作留空间。

## 十一、必须满足的体验底线

从本文件生效开始，后续任何设计与开发都必须满足下面这些底线。

用户无需理解技术术语就能完成核心操作。

用户无需安装开发工具就能使用系统。

用户无需学习新逻辑就能看懂主表格。

用户可以按日、周、月、年、自定义区间查看与导出结果。

用户可以手工补录，也可以批量导入。

用户可以补做历史区间日报。

用户可以在结果层快速核对、追溯、导出、打印。

用户可以在不懂数据库的前提下完成备份恢复。

用户在大部分时间里感受到的是“熟悉”和“省事”，不是“学习”和“配置”。

## 十二、明确禁止的设计倾向

后续设计与开发中，禁止出现下面这些倾向。

为了技术先进而牺牲用户上手门槛。

为了追求桌面壳而过早引入复杂多运行时架构。

为了追求炫酷 UI 而削弱表格主战场。

为了追求 AI 感而让 AI 越权做账务判断。

为了追求极简开发而把后续扩展空间堵死。

为了追求所谓灵活性而让用户到处自己配复杂规则。

为了节省几步开发工作，把导出打印、回查追溯、异常留痕这些关键能力砍掉。

## 十三、对 Claude Code 与后续开发者的直接约束

Claude Code 在阅读项目需求时，必须始终记住以下事实。

第一，这个项目的第一责任对象是中国财务人员，不是程序员。

第二，这个项目优先服务执行效率、核对效率、导出效率、汇报效率。

第三，界面、交互、技术选型、AI 设计，都必须服从用户的真实操作习惯。

第四，凡是用户看不懂、学不会、懒得配、不愿付费的设计，默认都不是好设计。

第五，任何模块设计、页面设计、技术决策，都应先问一句：这群财务人员今天真的会这样用吗。

## 十四、结论

“账房先生”服务的不是追求酷炫工作流的人群。

它服务的是每天在 Excel、网银、财务软件、日报、导出、补录、核对里来回切换的中国财务人员。

他们需要的是一个理解他们工作方式的软件。

他们不需要被教育成工程师。

他们也不需要被迫接受脱离现实的理想化系统。

这份文档自发布起，作为后续所有设计、开发、评审与取舍的前置约束。
