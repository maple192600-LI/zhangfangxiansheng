# 账房先生 — 用户需求与反馈记录

> 记录开发过程中用户提出的所有需求、反馈、修改方向和约束。按时间线整理，标注状态和优先级。

## 一、项目总体要求

### REQ-001: 产品定位
- **需求**: 面向中国财务人员的本地部署资金工作台
- **约束**: V1 仅交付"出纳资金日报最小可用闭环"
- **用户**: 不需要命令行操作，双击 start.bat 即可使用

### REQ-002: 技术栈锁定
- **需求**: Python/FastAPI 后端 + Vue 3 前端 + SQLite 数据库
- **约束**: 未经批准不得更换主技术栈
- **备注**: OCR 用 PaddleOCR，数据处理用 Polars/openpyxl

### REQ-003: 全中文界面
- **需求**: 全中文 UI，错误信息中文可读
- **约束**: 所有面向用户的文字必须是中文

## 三、报表模板管理

### REQ-005: 报表模板管理功能 (核心需求)
- **需求**: 开通报表模板管理功能，用户上传 Excel 模板定义报表格式
- **关联报表** (11种): 基础数据表、资金日报、现金日记账、账户余额表、收入明细表、支出明细表、主要账户余额表、月末盘点表、资金周报、资金月报、资金年报
- **核心逻辑**:
  1. 用户在"报表模板管理"页面为每种报表上传 Excel 模板
  2. 系统解析 Excel 表头，生成列配置
  3. 对应报表页面按模板格式展示数据
  4. 模板仅定义格式（列名、列顺序、列宽、分组方式），不涉及计算公式
- **状态**: 已实现（模板管理 + Excel 解析 + 6个已有页面接入 + 5个新报表页面）
- **优先级**: CRITICAL

### REQ-006: 报表导航结构
- **需求**: 不只是开通一个模板管理窗口，而是：
  - 报表模板管理作为独立页面
  - "资金日报表"子目录下：资金日报
  - "资金综合报表"子目录下：基础数据表、现金日记账、账户余额表、收入明细表、支出明细表、主要账户余额表、月末盘点表、资金周报、资金月报、资金年报
  - 模板管理与报表页面联动：上传模板后对应页面立即生效
- **状态**: 已实现

## 四、数据关联与准确性

### REQ-007: 基础数据表是唯一流水事实底座
- **需求**: 基础数据表是系统唯一流水事实底座，所有报表（现金日记账、日报、余额表等）都从基础数据表中提取数据
- **约束**: 数据流向为单向 — 基础数据表 → 各报表，报表不独立存数据
- **用户反馈**: "基础数据表是数据池，提取需要的数据来完成报表内容的填写"
- **优先级**: CRITICAL

### REQ-008: 各模块高度关联
- **需求**: 所有模块以基础数据表为数据池，提取需要的数据完成报表内容
- **用户反馈**: "这几个模块和基础数据表都是高度关联的，他们以基础数据表为数据池"
- **优先级**: HIGH

---

## 五、现金日记账详细需求

### REQ-009: 现金日记账查询口径 (用户明确指定)
- **需求**: 现金日记账的查询口径为**月份**
- **展示格式**:
  - 按月分块展示
  - 每个块包含：账户信息区（板块、核算方式、开户行、账户信息、银行编号）+ 明细表格
  - 账户信息区使用合并单元格
  - 明细区域不使用合并单元格
  - 字段：月、日、期初余额、摘要、收入、支出、余额
- **期初余额逻辑**: 取上月末余额，如无则取账户初始余额
- **模板上传后**: 即使无数据，也应显示表头
- **优先级**: HIGH
- **状态**: 部分实现（模板显示已实现，月份分块+账户信息区未实现）

## 六、功能验证要求

### REQ-010: 真实操作验证 (用户强烈要求)
- **用户反馈**: "你每次操作浏览器就出现这种弹窗"、"老子要的是真实不是模拟"
- **要求**:
  1. 必须用真实浏览器操作验证，不能用 curl 模拟
  2. 上传 Excel → 检查解析结果 → 对应页面查看表头 → 确认完整流程
  3. 重启后再次验证数据持久化
  4. 不让用户去测试，开发者必须自己验证
- **优先级**: CRITICAL
- **状态**: 已用 Playwright headless 浏览器验证通过

### REQ-011: 功能审查与代码审查
- **需求**: 对所有改动进行全面功能审查和代码审查
- **要求**: 找出未完成部分和代码错误，继续优化修正
- **用户反馈**: "把你刚才改动的这些内容和添加的模块进行一次功能审查和代码审查"
- **优先级**: HIGH

## 七、数据持久化

### REQ-012: 所有数据必须持久保存
- **需求**: 上传的数据和所有配置都必须持久保存，除非用户自己删除
- **用户反馈**: "整个项目中上传的数据和所有都应该保持，除非用户自己删除，但是我怀疑你现在的整个项目很多板块根本无法实际存储，在刷新页面或者重启之后数据就会丢失"
- **验证结果**: SQLite 持久化正常，重启后数据保留。之前的数据丢失假象由 `__pycache__` 缓存问题导致
- **优先级**: CRITICAL
- **状态**: 已验证通过
