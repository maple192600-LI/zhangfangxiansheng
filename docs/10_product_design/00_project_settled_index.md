## 二、当前已经可以正式定下来的内容

经过目前的讨论，下面这些内容已经足够稳定，可以进入项目正式文档。

### 1. 第一阶段的开发目标

第一阶段仍然只做资金板块的 MVP。

重点仍然是出纳资金日报这一条主链路，包括：

- 主数据与账户初始化
- 网银流水导入
- 手工流水录入与上传
- 上传结果预览
- 手动维护
- 基础数据表
- 资金日报表与基础综合报表
- 导出、打印、日志、备份、回滚
- AI 配置与解析助手骨架

### 2. 首页定位

首页已经明确独立置顶。

首页定位为出纳打开系统后的总控工作台。

首页服务于今天要做什么、当前做到哪一步、哪里有异常、哪些动作最该先做。

首页不承担复杂处理功能，也不做领导汇报分层页。

### 3. 资金板块的信息架构

资金板块当前已经可以正式固定为三组：

- 工作台
- 资金日报表
- 资金综合报表

其中“工作台”继续拆为：

- 网银导入
- 手工流水
- 手动维护
- 上传结果预览

这条链路已经具备较强稳定性，可直接进入开发依据。

### 4. 手工流水机制

手工流水不能只支持“系统内逐条录入”。

手工流水必须同时支持两条路：

- 系统内快速录入
- Excel 手工总表上传

Excel 手工总表必须支持一张表内录入多个法人、多个手工资金载体、多个现金账户、多个无网银账户的发生额。

系统真正需要严格收边界的位置，在解析、识别、校验、入库层，不在录入层。

### 5. 手工模板方向

手工模板不再采用大面积合并单元格作为正式上传模板。

原有那种以合并单元格为主的表，更适合作为人工查看版、打印版、历史工作底稿。

真正给系统上传的模板，应采用“多主体手工总表 V2”方向：

- 一个文件内支持多个法人和多个账户
- 每一行是一笔业务
- 核心识别列固定
- 大面积合并单元格取消
- 小计与汇总交给系统生成
- 结果更利于 Python 稳定解析

### 6. 字段池与模板方案机制

手工流水不再使用“固定死表头”模式。

系统维护统一字段池。

模板只允许在字段池范围内选择字段。

字段分为：

- 核心字段
- 业务可选字段
- 系统字段

用户通过字段方案来选择本次模板显示哪些列、列顺序如何、保存成什么方案。

所有自定义字段，必须先进入字段池维护，再允许被模板方案引用。
