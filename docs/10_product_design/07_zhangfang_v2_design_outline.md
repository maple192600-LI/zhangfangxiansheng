# 账房先生｜吸收《财务AI产品需求与设计全记录》后的新版设计总纲

## 一、这份文档解决什么问题

这份文档用于在已有《资金日报系统需求文档 v3》《技术选型约束》《目标用户与使用习惯约束》《V1 开发任务总控文档》基础上，吸收《财务AI产品需求与设计全记录》中的有效内容，形成一份更适合当前阶段使用的新版总纲。

这份新版总纲要解决的，不是把旧文档简单合并，而是把项目重新校正到一条更稳的路径上。

这条路径必须同时满足四件事。

第一，保留产品级完整视野。

第二，守住 V1 的落地边界。

第三，保证技术路线不自相矛盾。

第四，让后续 Claude Code、Codex 或其他 AI coding 工具在有限上下文里也不容易跑偏。

## 二、项目当前的真实定位

账房先生当前不是一个泛泛的“财务 AI 工作台”概念产品。

它当前首先是一套面向中国出纳、资金岗、财务主管的本地化资金处理系统。

第一阶段唯一要打透的主链路，是围绕资金日报形成的完整闭环。

这个闭环包括主数据维护、账户初始化、银行流水导入、首次模板识别、手工补录、区间日报生成、结果查看、异常处理、导出打印、基础看板、日志回滚与备份恢复。

AI 在这个阶段承担的是辅助角色。

它主要用于初始化、模板识别辅助、规则草案辅助、配置维护辅助和低风险引导。

它不负责替代规则引擎做每日高风险账务判断。

## 三、吸收旧方案后，当前最重要的校正

### 1. 项目应该按“三层结构”理解

第一层是产品总蓝图。

这里讨论的是项目为什么存在、最终要长成什么样、将来会有哪些模块、为什么它不该沦为另一个数据孤岛。

第二层是当前阶段的正式边界。

这里讨论的是 V1 现在到底做什么，不做什么，哪些能力只预埋，哪些能力必须真落地。

第三层是开发执行文档。

这里讨论的是目录结构、数据库表、接口、页面、服务层、测试场景、验收标准。

旧的《全记录》最大的问题，不是内容不好，而是把这三层揉在了一起。这样写出来的文档很像一份完整的产品说明书，但对开发执行来说，边界会变得模糊。

新版总纲必须把这三层重新拉开。

### 2. 数据库路线必须彻底说清

吸收旧方案后，数据库策略必须明确成一句话。

V1 继续使用 SQLite。

原因很简单。

当前阶段目标是单机、本地、双击即用、低门槛、便于备份恢复。SQLite 更符合这个阶段的真实条件。

PostgreSQL 不应该在 V1 正式切入主链路。

它应当被定义为未来进入多人协作、共享部署、集中式数据库阶段后的升级路线。

所以，新版文档里不能再出现前面写 PostgreSQL，后面又写 SQLite 的混合表达。

正确写法应该是：

当前物理数据库为 SQLite。

业务访问统一通过 SQLAlchemy ORM。

未来若进入多人协作或集中部署阶段，再通过数据库适配层切换 PostgreSQL。

### 3. 项目概念核心要从“账户”升级为“资金载体 + 标准资金事件”

旧方案里最有价值的部分，是它把表结构写得比较具体，便于落地。

现有需求文档里更进一步，提出了“资金载体”“标准资金事件”“批次”“异常项”“日报记录”这些更稳定的业务对象。

新版设计应该把两者结合起来。

逻辑层上，核心对象应该是：

法人主体。

资金载体。

批次。

标准资金事件。

日报记录。

异常项。

物理层上，V1 使用 accounts、fund_events、parser_templates、daily_report_runs、operation_logs 等具体表承接（详见 30_contracts/20_database_schema.md）。

也就是说，V1 的数据库可以先是“可跑的物理实现”，但上位设计必须用“资金载体”和“标准资金事件”来统领。

这样后面加现金、票据、财务公司单据、保证金、受限资金时，系统不需要推倒重来。

### 4. 首页和日报处理台必须分开

旧方案对页面描述已经很细了，这是优点。

但新版设计必须更明确地区分“首页工作台”和“日报处理台”。

首页工作台的任务，是快速回答问题和汇报。

它面向重点账户、今日收支、低余额预警、异常数量、临时补录数量、当前结果状态。

日报处理台的任务，是处理工作。

它才是导入流水、手工补录、区间补做、重算、确认、导出、打印、看异常的主战场。

如果把首页也做成一个重操作页，用户打开系统就会进入混乱状态。

如果把日报页又承担首页职责，管理信息和处理流程也会互相打架。

所以，新版设计里必须把这两个页面定位彻底分清。

### 5. Agent 架构保留，Agent 范围收紧

旧方案对 Agent 目录和七个文件的设计是有价值的。

这一套 shared、SOUL、AGENT、USER、MEMORY、TOOLS、RULES、WORKFLOW 的骨架可以保留。

但 V1 必须收口。

V1 只正式落下：

shared 公共区。

master 总管 Agent。

parser-assistant 解析助手。

其他如 cashier-agent、expense-agent、user custom agent，在总蓝图里保留，在目录和配置层允许预留，不进入正式主链路。

否则开发资源会被分散。

### 6. 模板系统不能只理解为“银行模板 JSON 文件”

旧方案说解析器可热插拔，这个方向对。

但如果把模板能力理解成“往 parsers 目录里扔一个 JSON 文件”，那还是太窄。

新版设计里，模板应当至少有两层。

第一层是数据库中的 parser_templates 元信息和映射规则。

第二层是样本表头、样本文件特征、人工修订记录、命中历史。

这样系统才能在 UI 里可查看、可修订、可复用、可审计。

文件目录可作为缓存和调试存储存在，但正式生效规则要统一以数据库与服务层为准。

### 7. 手工录入不是补丁，它是正式主链路

这一点旧方案和现有需求文档是一致的，但新版总纲要把重要程度再抬高一层。

因为这正是你最看重的现实财务场景。

当天只有一两笔时，用户不能被迫回到银行下载流水。

日报已经做完后，又补来一笔，也不能要求用户重跑整批导入。

所以，新版设计中手工录入必须有三种身份。

第一，低频少量交易的主录入方式。

第二，导入之后的局部补录方式。

第三，现金、票据、手工账户等非网银来源进入日报链的统一入口。

## 四、当前认可的整体设计

### 1. 产品层

账房先生是一套以资金日报为起点，以资金台账为底座，以规则引擎为核心，以 AI 辅助为增强，以未来模块化扩展为方向的本地财务工作台。

它当前服务的是中国财务人员最真实、最碎、最容易重复劳动的一段工作。

它要先把“每天整理资金结果”这件事做稳，再谈更大的财务协同平台。

### 2. 架构层

当前采用本地运行架构。

用户双击启动程序。

本地后端服务启动。

浏览器自动打开本地页面。

前端负责可视化界面和交互。

后端负责文件读取、标准化、规则计算、日报生成、导出打印、日志、备份与 Agent 骨架。

数据库当前用 SQLite。

通过 ORM 层隔离底层数据库差异。

### 3. 业务层

系统所有来源数据最终都要进入统一的标准资金事件池。

来源包括：

银行流水导入。

手工补录。

现金类记录。

票据类记录。

其他未来资金来源。

日报、工作台看板、账户流水查询、打印导出，都从标准资金事件和日报结果口径中生成。

系统不应该让日报直接绑定某一类原始文件格式。

### 4. 页面层

页面层建议固定为以下结构。

首页工作台。

日报处理台。

导入流水。

手工录入。

账户流水与历史查询。

账户管理。

模板与规则管理。

异常处理。

AI 配置。

Agent 管理。

系统设置。

其中 V1 真正完成的重点页面是：首页工作台、日报处理台、导入流水、手工录入、账户管理、AI 配置、系统设置。

账户流水查询、模板规则管理、异常处理页在 V1 可做基础版。

### 5. 数据层

新版设计建议把数据结构分成“逻辑对象”和“物理表”两层去写。

逻辑对象包括：

法人主体。

资金载体。

批次。

标准资金事件。

日报记录。

异常项。

模板规则。

AI 配置。

Agent 配置。

操作日志。

V1 物理表建议包括：

divisions。

entities。

accounts。

parser_templates。

fund_events。

daily_report_runs。

custom_fields（V1 预留）。

custom_field_values（V1 预留）。

ai_configs。

agent_configs。

operation_logs。

system_configs（V1 预留）。

实际建表增加了 import_batches 和 manual_field_pool / manual_template_schemes，让批次、字段池和模板方案独立管理，异常通过 fund_events.abnormal_code 标记。

这会比旧方案更稳。

### 6. 技术层

技术层沿用当前已锁定路线。

后端为 Python 3.11+ + FastAPI。

前端为 Vue 3 + Vite + Ant Design Vue。

图表用 ECharts。

数据处理以 Polars + openpyxl 为主，必要时兼容 pandas。

数据库为 SQLite + SQLAlchemy。

OCR 用 PaddleOCR 作为后续模块预埋。

打包为本地可执行程序。

这里不再把 PostgreSQL 放进当前阶段正式技术栈。

这样文档之间的职责更清楚。

## 六、新版总设计方案

### 1. 项目名称与定位

账房先生当前阶段是一套本地部署的资金处理产品。

它以出纳资金日报为入口，以统一资金事件数据底座为核心，以规则引擎和人工确认兜底为基本运行方式，以 AI 辅助初始化和维护为增强能力。

它服务的第一对象，是中国财务人员和资金岗，不是技术人员。

### 2. V1 的唯一目标

V1 只完成“出纳资金日报最小可用闭环”。

用户应能在 Windows 电脑上双击打开系统，通过浏览器界面完成：

主数据维护。

账户期初余额初始化。

多家银行流水导入。

新银行模板首次识别。

手工表格式录入。

日期区间日报生成。

日报查看、筛选、搜索、导出与打印。

基础首页工作台查看。

AI 配置。

Agent 基础骨架。

日志、备份、恢复与导入批次回滚。

### 3. V1 不进入主链路的内容

V1 不正式进入以下主流程。

发票 OCR 正式业务流程。

费用审批主流程。

合同管理主流程。

用户自建 Agent。

多 Agent 协作。

多人共享部署。

PostgreSQL 正式启用。

桌面壳封装。

### 4. 核心业务对象

法人主体用于承接公司主体。

资金载体用于统一表示银行账户、手工账户、现金账户、票据账户、财务公司账户等会影响资金结果的对象。

批次用于承接每次导入或每次手工补录形成的来源集合。

标准资金事件用于承接所有来源数据的统一结构。

日报记录用于表示某个日期或某个区间的一次计算结果。

异常项用于承接所有不能自动安全进入正式结果的数据。

模板规则用于承接银行模板、摘要规则、归属规则、业务分类规则等系统中间层。

### 5. 核心处理流

用户先维护主数据和账户初始化信息。

随后进入日报处理台，选择某一天或某个日期区间。

用户可以导入银行流水，也可以直接手工录入。

系统将原始来源文件留档，并生成批次信息。

系统对导入文件执行模板识别、字段映射、归属识别推荐、标准化解析、规则命中、异常校验。

通过校验的数据进入标准资金事件池。

日报、账户流水、首页看板、导出打印，都基于标准资金事件和日报结果口径生成。

若后续新增交易，用户可继续补录并重算。

### 6. 账户归属与规则原则

所有归属判断必须遵守稳优先原则。

文件中有明确账号、抬头、户名、账户特征时，优先命中主数据。

同模板对应多个账户时，先收缩候选范围，再结合历史余额链、时间连续性、交易对手特征做推荐。

仍然不能唯一确定时，必须人工确认。

摘要生成必须以规则引擎为主，不让 AI 在运行期自由发挥。

### 7. 连续日报链原则

系统必须维持资金载体在时间维度上的连续性。

已确认日报的期末余额，默认作为下一有效日期的期初来源。

没有发生额的日期可以不显式生成日报，但账期逻辑不能断。

用户补做历史区间时，系统必须自动承接之前有效余额，并按真实日期生成连续结果。

### 8. 页面体系

首页工作台用于汇报、预警和快速查看。

日报处理台用于导入、补录、重算、确认、导出和打印。

导入流水页用于文件上传、模板识别、预览确认和正式写入。

手工录入页用于 Excel 式横向表格录入。

账户管理页用于维护大区、法人、账户和初始化余额。

AI 配置页用于配置模型服务。

Agent 管理页用于展示内置 Agent 状态。

系统设置页用于备份恢复、日志查看、回滚和基础配置。

### 9. 技术底座

后端固定为 Python 3.11+ 与 FastAPI。

前端固定为 Vue 3 + Vite + Ant Design Vue。

图表固定为 ECharts。

数据库 V1 固定为 SQLite。

数据库访问统一通过 SQLAlchemy。

OCR 仅作后续预埋，不进入当前主链路。

运行方式为本地后端服务加浏览器界面。

### 10. AI 与 Agent 边界

AI 主要用于首次初始化辅助、新模板识别辅助、规则草案辅助、用户引导与低风险解释。

AI 不负责替代系统做每日高风险归属、摘要定稿、去重决策和余额正确性拍板。

Agent 体系从 V1 开始预埋工作区隔离结构。

V1 内置 shared、master、parser-assistant 三类基础骨架。

### 11. V2 以后升级路线

当 V1 主链路稳定后，再进入 V2。

V2 优先完善现金、票据、财务公司单据等特殊来源的正式独立能力，并接入发票 OCR 相关模块。

V3 再进入费用审批、合同管理、用户自建 Agent 等更复杂能力。

只有当单机版在真实使用中暴露出协同和部署瓶颈时，才进入 PostgreSQL 与集中部署评估。

## 七、结论

吸收旧方案之后，项目整体并没有被推翻。

真正发生的变化，是我们把它从“一份很丰满的大产品说明书”，重新收束成了一套“上位设计完整、当前边界清楚、技术路线稳定、开发执行不容易跑偏”的正式方案。

你这个项目现在最怕的，不是想得不够多。

你现在最怕的是文档层级混乱，导致 AI coding 工具一边看着远期蓝图，一边误以为这些都要在 V1 一次做完。

所以新版总纲的核心价值，就是替你把脑子重新排成顺序。

先把出纳资金日报这条主链路彻底打透。

等这条链跑稳了，后面的票据、发票、费用、合同、Agent 平台，才有真正能长出来的根。
