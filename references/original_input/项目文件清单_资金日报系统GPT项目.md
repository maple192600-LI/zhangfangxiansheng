# 项目文件清单｜资金日报系统 GPT 项目

## 一、项目定位

这个 GPT 项目用于长期承载“资金日报系统”从概念、需求、规则、页面、模块、技术设计到开发落地的完整过程。

项目中所有聊天、文档、样本文件，都应围绕同一个目标持续沉淀，避免重复讨论、概念漂移和版本混乱。

## 二、项目中建议长期保留的文件

### 1. 主需求文档

这是整个项目最重要的核心文件，只保留当前生效版本。

建议文件名：
- 资金日报系统_需求文档_v当前版本.md

作用：
- 作为当前项目唯一主需求依据
- 后续模块设计、页面设计、开发任务都必须基于它展开

注意：
- 老版本不要删除，但要放进“历史归档”目录或统一加“已失效”标记
- 项目中任何新讨论，如果与主需求文档冲突，应优先修订主文档

### 2. 项目总指令

建议文件名：
- 项目指令_资金日报系统GPT项目.md

作用：
- 定义 GPT 在该项目中的角色、边界、工作方法、输出格式与长期原则
- 保证后续任何聊天都围绕同一套脑子工作

### 3. 银行流水样本文件

建议长期保留所有真实或脱敏后的样本文件，例如：

## 四、项目中的聊天线程建议

建议这个项目里长期固定几个主题聊天，不要什么都混在一条里：

### 1. 总需求与版本修订
专门讨论需求变更、场景补充、版本更新。

### 2. 页面与交互设计
专门讨论首页、日报处理台、账户流水页、打印导出等前端问题。

### 3. 模块与规则设计
专门讨论标准化解析、账户归属识别、规则字典、异常池等系统逻辑。

### 4. AI Agent 与初始化
专门讨论 AI Agent 负责什么、怎么做初始化、怎么辅助字典更新。

### 5. 开发任务与技术落地
专门把成熟文档继续拆成给 Claude Code、Codex 可执行的开发任务。

### 6. 测试与验收
专门整理测试用例、异常样本、验收标准。

## 五、建议的项目推进顺序

最稳的推进顺序如下：

先把当前生效需求文档放进项目  
再放项目总指令  
再放所有银行样本和模板  
然后开始拆模块设计  
再拆页面设计  
再沉淀规则字典  
再做开发任务拆解  
最后才进入具体实现和测试

不要一开始就把时间花在前端颜色和圆角上。先把业务脑子搭稳。

## 六、这个项目最重要的管理原则

第一，任何时候都尽量只保留一个当前主需求文档。  
第二，每次重要讨论结束后，都尽量沉淀成正式文件。  
第三，样本文件不要丢，它们是后续开发和测试的根。  
第四，页面、模块、规则、测试尽量分开沉淀，不要混着写。  
第五，后续所有扩展都要避免把这个系统做成新的数据孤岛。
