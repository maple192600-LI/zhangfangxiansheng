# 资金日报系统需求文档 v3

## 一、文档目的

本文档用于完整沉淀本项目在前期讨论中已经明确的业务背景、产品目标、核心原则、关键场景、功能范围、模块设计、页面逻辑、数据处理机制与实施路径。本文档的目标，是让开发者、AI coding 工具、协作人员在没有额外口头解释的情况下，也能准确理解本项目要解决什么问题、为什么要做、必须覆盖哪些真实工作场景、哪些能力必须预埋、哪些能力可以分阶段落地。

本项目面向的核心对象，是日常负责付款、收款、资金日报、银行日记账、现金台账、票据台账、手工资金记录整理的财务人员、出纳、资金岗人员。系统的目标，是把目前大量依赖 Excel、复制昨日表、手工粘贴流水、手工分类、手工校验余额、手工打印归档的低效流程，转化为一套连续、可追溯、可扩展、可分阶段实施的资金处理产品。

## 二、项目背景与业务痛点

当前很多企业的资金处理并不规范。付款和收款经常是碎片化发生的，上午一笔、下午一笔、临时又来一笔，很多业务没有固定批量处理节奏。出纳或资金岗在一天结束前，通常需要根据当天实际发生的付款和收款，整理银行流水、填写银行日记账、生成资金日报、核对余额，并将结果导出或打印给领导、老板或内部管理层使用。

现有典型工作方式通常依赖 Excel。每天需要复制上一日的日报模板，建立新一天的表，将上一日余额作为新的期初余额，再逐笔填入当天发生的收支，人工整理摘要、人工校验余额、人工保存文件、人工打印。这种方式在付款量很大时非常耗时，在付款量很小时同样不轻松，因为哪怕当天只动了一笔，也依然要走一整套重复动作。长期下来，效率低，连续性差，出错难追，归档混乱，历史回查成本高。

更复杂的是，一个集团或老板名下可能存在多个法人主体。多个不同法人主体可能都在同一家银行分别开立账户。比如 A 公司有中国银行账户，C 公司有中国银行账户，D、E、F 公司也都可能有中国银行账户。不同法人在同一家银行导出的流水模板可能完全一致，而有些流水文件中又未必带账号、公司名称等足以唯一识别账户归属的信息。再叠加一些资金载体无法从网银导出流水，只能手工维护，或者存在现金、票据、财务公司单据、保证金、手工账户等情况，最终都要影响日报结果。

因此，本项目需要解决的，不只是“读一份银行流水”，而是一整套围绕资金日报、银行日记账、现金与手工账户、票据与其他资金工具、连续账期、账户归属、手工补录、历史补做、导出打印、规则维护与后续扩展展开的资金处理问题。

## 三、项目定位

本项目定位为一套本地运行、规则驱动、可留痕、可追溯、可分阶段实施、以资金日报和资金台账为核心结果的资金处理产品。

系统前台在交互上可以保留较强的 Excel 使用习惯，因为财务人员天然更熟悉表格视图、按列查看、筛选、连续余额、逐笔核对的方式。系统底层必须具备明确的数据来源管理、规则引擎、连续账期承接、异常校验、重算机制、模块扩展能力。

系统最终服务的是日常工作效率、领导汇报、流程标准化、未来数据分析与后续产品扩展双重甚至多重目标。

它既要让出纳和资金岗在真实混乱的工作环境里，尽可能快地完成日报生成与台账整理，也要支持首页看板快速回答老板和领导最常问的资金问题。进一步看，它还要避免成为新的数据孤岛，为后续扩展付款计划、待办、分析、预警、票据、共享协作等功能留出结构空间。

## 四、项目愿景

本项目不是只做一个日报生成小工具，也不是只做一个银行流水解析器。

它的目标，是在中小企业场景中，把过去需要大量人力做的数据搬运、流水整理、日报编制、余额衔接、票据跟踪、手工台账维护，逐步沉淀成标准化、产品化、可复制的能力。未来理想状态下，可以支撑两三个人完成过去十几个人才能完成的一部分基础资金整理与台账处理工作，把人从重复录入和反复搬运里解放出来，转向更有价值的分析、判断和管理动作。

同时，这个产品也应该允许个人用户或单个企业只购买和使用其中某一个模块。即便只使用一个模块，也应该能显著提升效率，而不是强迫用户一次性上整套系统。

## 五、核心目标

本项目要实现以下核心目标。

第一，减少 Excel 重复劳动。将原本每天复制上一日表格、粘贴期初余额、录入发生额、校验余额、导出打印的流程压缩为更短路径。

第二，支持两类极端场景同时成立。付款笔数多的时候，支持批量导入流水、自动解析、自动生成日报。付款笔数少的时候，支持快速手工录入一两笔，重算日报后立即生成结果，无需为了少量业务再去银行下载流水。

第三，支持连续账期。系统必须自动承接上一期已确认日报的期末余额，作为下一期的默认期初余额来源。即便中间某几天没有发生额，也不能让账期断裂。用户再次做日报时，系统要自动从最后一次有效期末余额继续往下接。

第四，支持历史补做。用户如果因出差、请假或其他原因漏做了几天日报，系统必须允许按日期区间补做，并自动按日期串联生成连续日报链。

第五，支持局部补录。若日报已经做完，后续又发生一笔临时紧急付款，用户应该能直接在某个账户或某日的日报下快速手工补录这一笔，然后重算当天日报，而无需重新完整导流水再走一大遍流程。

第六，支持复杂账户归属场景。面对多个法人、多个银行、多个同模板账户、部分流水缺少账号和户名、部分资金只能手工录入的现实情况，系统要通过主数据、模板识别、历史余额链、来源特征和人工确认完成归属识别，绝不能在信息不足时盲目自动入账。

第七，运行期尽量不依赖 AI 自由发挥。资金数据敏感，日常处理必须依靠确定性脚本、配置、规则与校验。AI Agent 的价值主要体现在首次初始化、后续规则字典更新、模板特征维护、模块配置辅助、用户引导与低频复杂设置，而不是每天代替系统核心规则做高风险判断。

第八，首页可用于汇报。工作台必须能展示重点资金看板，支持重点账户筛选、自定义展示字段，便于快速回答老板和领导问题。

第九，产品整体必须可扩展、可拆分、可分阶段实施，既能支持整套产品落地，也能支持单模块独立使用。

## 六、基本原则

本项目日常运行坚持规则优先、校验优先、人工确认兜底、模块可扩展、阶段可实施的原则。

凡是能够通过模板、字段、公式、余额链、规则引擎解决的，都应使用工程化手段完成。凡是归属不确定、字段缺失、余额对不上、摘要无法可靠生成、来源可能重复的数据，都必须进入异常处理或人工确认流程，不能静默自动入账。

系统在视觉和交互上应尊重财务习惯。日报和各类台账页应以表格为主，保留强烈的 Excel 感与可筛选、可冻结、可连续查看的体验。

本项目的模块设计要一次性想全，但项目实施可以分步推进。也就是说，银行流水、手工账户、现金、票据、财务公司单据等都应在产品设计层面被纳入统一框架，但实际开发可以按优先级一模块一模块上线、测试、修正、再扩展。

## 七、AI Agent 功能定位

AI Agent 应作为产品内的辅助型能力存在，而不是核心账务判断引擎。

AI Agent 在首次使用时，应帮助用户完成初始化工作，例如理解企业当前有哪些法人主体、有哪些银行和账户、有哪些重点账户、现有日报口径是什么、摘要和字典规则如何配置、哪些来源需要手工维护、哪些来源可以批量导入。

在后续使用过程中，当用户的账户体系、摘要规则、业务类型、票据种类、模板特征发生变化时，AI Agent 可以帮助用户快速更新规则字典、识别新模板的差异、辅助生成新的解析配置、引导用户完成新模块启用。

AI Agent 还可以在低频复杂场景中充当“初始化和维护助手”，例如引导用户增加新的法人主体、创建新的资金载体、初始化票据台账字段、梳理某类费用摘要规则、生成某银行模板的初始解析配置。

AI Agent 不应用于在运行期直接替代系统进行每日高风险账务判断。每日流水的归属、摘要、余额衔接、标准化、日报生成，仍应以可追溯的工程规则和人工确认机制为主。

## 八、目标用户与使用角色

本项目的核心使用者，是出纳、资金岗、财务人员、财务主管。

直接操作系统的，一般是出纳或资金岗。他们负责上传流水、手工补录、检查异常、重算日报、确认结果、打印和导出。

需要查看结果和听汇报的，一般是财务主管、老板、管理层。他们未必每天自己操作系统，但会频繁询问重点账户余额、今日已付金额、重点法人资金情况、某些账户是否紧张等问题。因此首页看板要支持快速汇报使用。

在系统配置和迭代阶段，可能还存在一个“配置维护角色”，由懂业务的人或管理员负责维护规则、字典、模板和模块启用状态。AI Agent 应优先帮助这一角色降低初始化与维护成本。

## 九、产品模块总图

本项目应按产品级能力设计完整模块，而不是只围绕银行流水做局部设计。整体建议包含以下模块。

主数据中心模块，用于维护法人主体、银行、银行账户、手工账户、现金账户、票据账户、财务公司账户、重点账户组等。

银行流水处理模块，用于导入银行流水、模板识别、字段映射、标准化解析、摘要规则处理、日报结果生成。

手工资金记录模块，用于处理那些无需或无法导流水的低频交易、零散交易、补录交易、临时调整交易。

现金处理模块，用于维护现金类资金载体、现金收支记录、现金余额衔接，并将其纳入日报链和资金汇总。

票据处理模块，用于维护银承、商承、财务公司票据、保证金、受限资金等类型的记录、流转、状态变化及其对日报和资金视图的影响。

账户归属识别模块，用于解决跨法人、同银行、同模板条件下的归属判断问题。

规则与字典模块，用于维护摘要规则、业务类型规则、模板特征、费用分类、来源标记等。

异常处理与重算模块，用于接住所有不确定、对不上、可能重复或需要人工确认的数据，并支持重算。

首页工作台与汇报看板模块，用于展示重点账户和重点指标，方便领导询问与日常汇报。

日报处理台模块，用于完成导入、补录、区间补做、重算、确认、导出打印等操作。

账户流水与历史查询模块，用于回查某个法人、某个账户、某个时间区间内的明细和结果。

导出打印模块，用于 Excel 导出、打印版式输出、归档留存。

AI Agent 辅助模块，用于初始化、配置维护、规则更新辅助、模板更新辅助和用户引导。

以上模块都应进入产品总体设计。项目实施时可以按优先级逐个落地，例如先完成银行流水模块，再接手工和现金，再接票据和更复杂的扩展。

## 十、第一阶段建议范围

第一阶段优先建议完成以下能力闭环。

主数据中心中的法人、银行、银行账户、重点账户组。

银行流水处理模块中的导入、识别、解析、标准化、归属、规则处理、日报生成。

手工资金记录模块中的快速补录、低频手录、临时调整录入。

日报处理台、首页工作台、导出打印、异常重算。

连续日报链与历史区间补做。

AI Agent 的初始化与规则更新辅助。

现金、票据、财务公司单据等模块在第一阶段就要预埋模型、页面入口和数据结构设计，但可在银行模块稳定后分阶段继续实施。

## 十一、核心业务对象定义

### 1. 法人主体

系统中每个公司、每个法人单位都应作为独立主数据存在。

### 2. 资金载体

资金载体是一个总概念，用于统一管理所有会影响日报和资金结果的对象。资金载体至少包括银行账户、手工账户、现金账户、票据账户、财务公司账户等类型。

### 3. 账户主数据

每个银行账户、手工账户、现金账户或其他资金载体，都必须挂接到某个法人主体下，并具备自己的名称、简称、所属银行、币种、状态、重点标记、余额管理方式等属性。

### 4. 日报记录

日报记录是系统中某一天某个口径下的一次资金结果快照。它有自己的日期、状态、版本、覆盖范围、结果数据和来源追溯。

### 5. 批次

批次指用户某次导入的一份或一组原始文件，也可以指一次手工补录形成的来源集合。批次必须有状态、来源类型、归属、处理记录和留痕。

### 6. 标准资金事件

所有银行流水、手工补录、现金收支、票据变动、后续其他来源数据，最终都要进入统一的标准资金事件结构。日报和各类台账都基于标准资金事件生成，不直接基于原始文件计算。

### 7. 异常项

凡是归属不确定、模板识别不确定、余额对不上、字段缺失、可能重复、摘要无法可靠生成的记录，都要进入异常项。

## 十二、首页工作台要求

首页应作为工作台存在，并保留数据看板能力，服务于快速查看与汇报场景。

首页应支持自定义展示内容。用户可以配置自己关注的重点法人、重点账户、账户组或关键指标。

首页建议支持展示的核心信息包括：重点账户余额、上日余额、当前口径余额、今日已收、今日已付、法人维度资金汇总、低余额预警、今日临时补录笔数、未处理异常数量、已纳入最近一版结果的账户范围等。

首页应优先帮助用户快速回答老板或领导的高频问题，而不是承担复杂操作功能。

系统需要支持“重点账户组”概念，允许用户从大量账户中筛选出日常最关心的一组主账户进行看板展示。

首页可支持用户保存个性化看板方案，例如“老板汇报视图”“我自己的工作视图”“重点账户视图”等。

## 十三、日报处理台要求

日报处理台是系统真正的主战场，承担实际工作流处理。

用户进入日报处理台后，应先选择某一天或某个日期区间作为当前处理对象。

页面需能展示当前日报任务已纳入的法人、账户、来源批次、手工补录数量、异常数量和当前结果状态。

页面需支持“导入流水”“快速补录”“批量重算”“确认结果”“查看异常”“导出 Excel”“打印”等高频动作。

页面中应有批次列表区，用于展示当天或当前区间已导入的各类来源数据，包括银行流水批次、手工批次、现金记录批次、票据批次等。

页面中应有日报预览表区，用于展示当前重算结果。该表应保持强烈的 Excel 表格感，支持筛选、冻结列、分组、搜索、跳转来源明细。

页面中应有明细追溯能力，允许用户点开某一笔或某一行结果，查看其对应的原始来源、标准化结果、摘要规则命中、校验结果和修改记录。

页面中必须覆盖手工维护类资金载体的处理能力。也就是说，手工账户、现金、票据、财务公司单据等，不应被排除在日报处理台之外，而应作为日报链中的正式来源参与重算与结果生成。

## 十四、标准资金事件结构要求

系统必须定义一套统一的标准资金事件字段，用于承接所有来源数据。至少应包含以下信息：交易日期、交易时间、币种、收入金额、支出金额、签名金额、交易后余额或状态余额、对方户名、对方账号、业务类型、业务名称、用途、摘要原文、备注、来源批次、来源类型、所属法人、所属资金载体、是否手工、是否待回补、校验状态、异常标记等。

所有来源数据进入系统后，都要尽可能先落成这一结构，再进行后续日报生成和规则处理。

## 十五、账户归属识别要求

账户归属识别是本项目的高价值能力之一，必须单独设计，不能混在解析器内部偷偷处理。

系统必须支持以下归属判断方式。

当文件中出现账号、账户名称、对账单抬头、开户地址、账户简称等唯一信息时，系统应优先直接命中主数据。

当文件模板明确指向某个银行或某类来源时，系统应先将候选账户范围缩小到该银行或该类账户。

当文件缺少明确归属信息时，系统允许基于历史余额链、最近一次已确认期末余额、交易对手特征、常见摘要、时间连续性等进行推荐归属，并给出置信度。

当候选仍然不唯一时，系统必须要求人工确认，确认结果应被保存，作为后续同类文件的历史参考。

系统严禁在归属信息不足时强行自动入账。

## 十六、摘要生成与规则引擎要求

日报和各类台账中的摘要必须符合财务使用习惯，能够被人快速理解和追溯，不允许简单照搬所有原始字段，也不允许让 AI 在运行期自由发挥。

摘要规则至少要支持以下逻辑。

对于手续费、服务费、短信费、回单费等费用类交易，应统一按规则映射为“付款手续费”或更细的费用分类。

对于结息、收息、利息类交易，应统一识别为“结息收入”或相应规则名称。

对于贷款放款、贷款回收、贷款归还等业务，应按规则输出统一摘要。

对于外部来账或往账，优先依据对方户名、业务名称、用途、摘要原文等字段生成摘要。

对于内部转款场景，当交易双方都属于己方账户体系时，应按规则生成“内部转款”或更具体的用途说明。

现金、票据、财务公司单据等来源，也应能通过规则引擎输出统一风格的摘要和业务标签。

当摘要无法可靠命中时，允许进入人工修正，修正结果应可沉淀为后续规则。

## 十七、连续日报链与期初期末承接要求

本项目必须具备“连续日报链”能力，这是区别于单纯表格录入工具的核心要求之一。

系统应自动维护每个法人、每个资金载体在时间维度上的日报连续性。已确认日报的期末余额，默认作为下一有效日期日报的期初余额来源。

若中间某几天没有发生额，系统不强制要求用户手工生成空白日报，但账期在逻辑上必须连续。用户下一次再做日报时，系统应自动从最后一次有效期末余额承接，不允许账期割裂。

若用户漏做了若干天日报，系统必须支持按日期区间补做，并依次根据前一有效日报结果自动承接和计算。

## 十八、日报生成方式要求

日报生成必须支持多种实际工作路径，不能只服务某一种理想化处理习惯。

### 1. 批量导入模式

适用于当天业务笔数较多的情况。用户在一天结束前，或某个阶段性收口时，批量导入若干银行流水文件，系统解析、识别、归属、标准化后，生成当天日报草稿或结果。

### 2. 快速手工补录模式

适用于当天业务极少的情况。若当天只有一笔或两笔交易，用户不想再去银行下载流水上传，系统必须允许直接手工快速录入这几笔交易，然后重算生成当天日报。

### 3. 局部补录模式

适用于日报已经基本完成或已做出一版结果后，临时又新增少量交易的场景。用户应可直接在某天某个账户下新增一笔或几笔手工交易，再点击重算，将其纳入当天日报，而无需重新走完整导入流程。

### 4. 历史区间补做模式

适用于用户漏做了若干天日报、出差或请假后回来补账的场景。系统应支持选择某个日期区间，导入这段时间内的流水文件或手工录入交易，按日期自动归集和重算，生成连续日报结果。

### 5. 当天无发生额承接模式

若某一天没有任何发生额，系统可以允许用户不显式生成当天日报，但下次生成日报时，必须自动承接最后一次有效日报的期末余额作为新一期期初余额。

## 十九、手工、现金、票据等其他资金类型处理要求

手工录入、现金、票据、财务公司单据等都属于产品正式能力，不应被当作边缘补丁或遥远扩展。

手工录入至少应支持录入法人、资金载体、交易日期、交易时间、摘要、收入、支出、余额、对方单位、备注、来源说明等。

现金处理至少应支持现金账户维护、现金收支录入、现金余额连续承接、现金日报纳入。

票据处理至少应支持票据基础信息维护、状态跟踪、到期信息、金额变化、对日报和资金视图的影响记录。

财务公司单据等特殊来源应能以统一资金事件的方式进入系统，并被纳入日报重算和汇总逻辑。

这些模块在项目实施上可以分阶段完成，但在产品设计上必须从一开始纳入整体框架。

## 二十、异常处理要求

系统必须提供异常处理机制，且异常不能被掩盖。

以下情况应进入异常池：模板识别不确定、账户归属不确定、期初期末衔接失败、滚动余额不一致、借贷汇总和明细不一致、关键字段缺失、摘要规则未命中、疑似重复入账、手工补录与后续正式来源疑似重复等。

异常池应支持按日期、法人、账户、批次查看异常，支持人工修正、重新归属、重新套用规则、重算结果，并保留处理留痕。

## 二十一、账户流水与历史查询要求

系统应提供账户流水或资金明细查询页面，用于查看某个法人、某个账户、某段时间内的全部标准化交易与结果记录。

该页面用于历史回查、问题追溯、来源确认、余额连续性检查，不应只局限于当天日报。

## 二十二、导出与打印要求

系统必须支持导出 Excel 和打印功能。

日报生成后，用户应可一键导出为 Excel，用于继续内部加工、备份或外部传递。

系统还应支持打印视图或打印模板，便于纸质归档、签字流转或向领导汇报。打印结果应尽量接近财务习惯中的日报格式，而不是简单打印网页界面。

## 二十三、数据流与处理流说明

系统整体处理流应如下。

用户先维护法人、账户、重点账户组、规则、现金载体、票据载体等主数据。随后，用户在某一天或某个日期区间下，导入流水文件或直接进行手工录入。系统先将文件留档，再做规范化、模板识别、归属识别、标准化解析、规则引擎处理、异常校验。通过校验的数据进入统一标准资金事件池。日报、日记账、现金视图、票据影响视图等均从标准资金事件池计算生成。用户可随时重算，查看预览结果，并在确认后生成正式结果。若后续有新增交易，可再次补录和重算。导出和打印均基于当前结果视图完成。

## 二十四、技术实现原则

系统应优先使用本地部署、本地文件处理和可审计脚本方案。推荐使用 Python 等确定性脚本语言构建核心引擎，避免运行期大量依赖 AI 模型。

文件读取、模板识别、标准化、规则引擎、余额校验、重复检查、导出打印等功能，应全部设计为可追溯、可重跑、可测试的工程模块。

AI Agent 可以在开发期和配置维护期辅助写解析器、辅助设计规则、辅助整理模板特征、辅助初始化和规则更新，但运行期不承担每日高风险账务判断。

## 二十五、分阶段实施建议

产品设计应一步想全，实施节奏可以逐步推进。

第一阶段，完成银行流水主流程、首页工作台、日报处理台、快速手工补录、导出打印、异常重算、连续日报链。

第二阶段，完善手工账户、现金账户、低频零散资金记录。

第三阶段，接入票据、财务公司单据、保证金等特殊资金来源。

第四阶段，进一步增加分析、预警、付款计划、跨模块协同等能力。

各阶段之间应共用统一的主数据、标准资金事件模型和结果投影逻辑，避免形成新的数据孤岛。

## 二十六、避免数据孤岛的产品要求

本项目在设计上必须考虑未来与其他功能的衔接，避免再造一个只能单独看日报、无法与其他资金管理能力协同的数据孤岛。

因此，核心数据模型、主数据、标准事件、批次、规则、结果视图都要具备可扩展性。后续无论增加付款计划、待付款清单、分析看板、跨法人汇总、票据管理、共享协作，最好都能在同一数据底座上继续扩展，而不是另起炉灶。

## 二十七、项目成功标准

本项目成功的判断标准，不在于页面多炫，也不在于模板支持数量有多少，而在于它是否真正让日常资金日报工作变快、变稳、变连续、变可追溯。

当付款量大时，用户应能通过批量导入在较短时间内完成日报生成。

当付款量小时，用户应能通过快速补录在数分钟内完成当天日报。

当漏做了几天日报时，用户应能通过区间补做快速补齐，并自动承接期末余额。

当日报做完后临时新增一笔付款时，用户应能快速补录并重算，而不必重走完整流程。

系统必须让结果更连续、回查更容易、打印导出更方便、汇报更直接，并为后续模块化扩展留出空间，最终显著优于原本每天复制 Excel、手工录入和手工核对的旧方式。
