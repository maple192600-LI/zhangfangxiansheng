## 一、产品构思与起源

### 1.1 问题背景

出纳日常工作的核心痛点：

- **多法人、多银行账户管理**：一个出纳可能同时管理多个法人主体（集团公司），每个法人名下又有多个银行账户（基本户、一般户、临时户），每天需要汇总所有账户的收支情况
- **银行流水格式不统一**：各家银行网银导出的流水文件格式完全不同——列名、列数、排列顺序、文件格式（Excel/CSV/PDF）各异
- **部分账户无网银**：贷款账户、新开户未满期账户等无法导出电子流水，需要手动录入
- **非现金支付工具**：银行承兑汇票、商业承兑汇票、信用证等有价凭证也需要记录
- **手工日报效率低**：每天要从各家网银分别下载流水，手工复制粘贴到Excel日报模板中，重复劳动量大
- **筛选查询不便**：按日/周/月/年、按账户、按法人单位等维度的查询依赖Excel手工筛选

### 1.2 产品定位

**"账房先生"（ZhangFang）** — 一个面向中国财务人员的本地化AI工作台。

核心特征：

- **toC产品思维**：不是企业级SaaS，而是给单个财务人员自己用的工具
- **本地部署**：不依赖云服务器，双击EXE即用，数据完全在本地
- **工程编排为主，AI辅助**：能用确定性代码解决的不调AI（省token、保准确），AI用于解析器创建、数据校验、智能查询等场景
- **模块化架构**：出纳资金日报只是第一个模块，后续可逐步加入发票识别、费用审批、合同管理等功能
- **非技术用户友好**：目标用户连CMD都不会打开，一切操作必须在可视化界面上完成

### 1.3 目标用户画像

- 中国公司的出纳、会计、财务经理
- 日常主要使用Excel和金蝶/用友等财务软件
- 无编程基础，无技术背景
- 操作系统：Windows 10/11
- 电脑性能：普通办公电脑
- 第一版：单人使用

### 1.4 开发路径规划

| 阶段  | 模块                          | 说明                                                 |
| --- | --------------------------- | -------------------------------------------------- |
| 当前范围  | 出纳资金日报                      | 多法人+多银行账户的资金日报自动汇总                                 |
| 后续范围  | 发票OCR识别+分类                  | GitHub开源OCR识别项目内置，同样能用于费用审批和合同审批的场景。不同类型发票，按规则提取信息 |
| 远期范围  | 费用审批流程                      | 付款申请单审核、费用报销审核                                     |
| V4  | 合同审批                        | 合同管理、执行跟踪                                          |
| 后续  | 凭证模板生成、报表统计、自定义模板、用户自建Agent | 远期愿景                                               |

## 二、技术选型与架构

### 2.1 技术栈

| 层级   | 技术                      | 选择理由                                         |
| ---- | ----------------------- | -------------------------------------------- |
| 后端   | Python + FastAPI        | 处理Excel/CSV能力最强（pandas、openpyxl），AI SDK生态最丰富 |
| 前端   | Vue.js + Ant Design Vue | 蚂蚁集团开源UI组件库，专为企业级应用设计，天然支持中文                 |
| 数据库  | PostgreSQL              |                                              |
| OCR  | PaddleOCR（Python包）      | 集成到后端，不依赖外部工具                                |
| 打包   | 免安装整合包                  | 项目内置虚拟环境，完成依赖，模型的下载和安装，用户通过bat脚本启动           |
| AI接入 | 统一接口层                   | 支持智谱清言、Kimi、通义千问、OpenAI、本地Ollama等多厂商         |

### 2.2 整体运行机制

```
用户双击 ZhangFang.bat启动
    ↓
启动本地 Python 服务（FastAPI，端口 19527）
    ↓
自动打开默认浏览器 → http://localhost:19527
    ↓
用户在浏览器中操作（和普通网站一样）
    ↓
关闭浏览器窗口 → 后台服务自动退出
```

**端口策略**：默认使用19527端口，如果被占用则自动递增尝试19528、19529...

### 2.4 核心设计原则

1. **模块完全独立**：删掉 modules/invoice/ 整个文件夹，其他模块不受任何影响
2. **核心不动**：core/ 里的代码开发完成后基本不需要改，只提供通用能力
3. **解析器可热插拔**：新增一家银行的解析器，就是在 parsers/bank_templates/ 里加一个JSON文件
4. **配置集中管理**：所有AI密钥、数据库路径等配置集中在config.py，通过界面修改
5. **虚拟环境隔离**：开发用venv，打包后完全独立，不污染用户电脑环境
6. **安装位置可选**：不锁死C盘，用户可选择任意磁盘安装

## 三、数据模型与数据库设计

> SQLite 单文件数据库，通过 SQLAlchemy ORM 操作。当前使用 SQLite（零配置），将来需要多人协作时可一行配置切换 PostgreSQL。

### 3.1 表间关系总览

```
大区/板块 divisions（可选）
    └── 1:N → 法人单位 entities
                  └── 1:N → 银行账户 accounts
                                │  （银行存款/现金/承兑/信用证...）
                                │  （网银导入 或 手动填写）
                                └── 1:N → 交易流水 transactions
                                              │
                                              └──→ 汇总到「资金日报」

解析器模板 parser_templates ← 被银行账户使用
自定义字段 custom_fields ← 扩展交易流水字段
自定义字段值 custom_field_values ← 存储扩展字段的值
AI配置 ai_configs ← 全局AI服务配置
Agent配置 agent_configs ← 各Agent角色配置
操作日志 operation_logs ← 审计追踪
```

## 四、账户管理体系

### 4.1 账户层级结构

系统采用三级层级管理：

```
大区/板块（用户自定义，可选）
  └── 法人单位
        └── 银行账户
              └── 交易流水
```

- **大区/板块**：可选层级。大型集团可设"华北大区""华中大区"或"抖音事业部""淘宝事业部"。小公司不用就留空
- **法人单位**：必须层级。对应一个独立的法律实体公司
- **银行账户**：必须层级。包括银行存款账户、现金账户、承兑汇票账户等

### 4.2 账户类型

| 类型 | 说明 | 界面显示 |
|------|------|---------|
| basic | 基本存款账户 | 基本户 |
| general | 一般存款账户 | 一般户 |
| temporary | 临时存款账户 | 临时户 |
| cash | 现金账户 | 现金账户 |

> 界面全部显示中文，数据库内部用英文代码方便程序处理。用户永远不会看到英文代码。

### 4.3 支付工具（instrument_type）

| 工具类型 | 说明 |
|---------|------|
| 银行存款 | 普通银行转账 |
| 现金 | 纸币收付 |
| 银行承兑汇票 | 银行担保的商业汇票 |
| 商业承兑汇票 | 企业担保的商业汇票 |
| 信用证 | 银行开立的付款承诺 |
| 其他 | 未来可能的新型支付工具 |

这些类型不写死，系统提供增删功能，用户可根据实际业务添加新的支付工具类型。

### 4.4 数据录入方式分类

根据账户是否能导出电子流水，分为两种录入方式：

| 录入方式 | 适用场景 | 操作方式 |
|---------|---------|---------|
| 网银导入 | 有网上银行的账户 | 从网银导出Excel/CSV → 上传到系统 → AI解析写入 |
| 手动填写 | 无网银的账户（贷款账户、新开户等） | 在系统界面上直接选账户 → 填写流水 → 提交 |

**关键设计**：不管数据怎么来的（导入还是手填），最终都写入同一张 transactions 表，数据结构完全一致。区别只是录入方式不同（批量导入 vs 手动填写）。

### 4.5 账户管理界面

侧边栏 → "设置" → "账户管理"

**树形展示**：
```
├─ 华北大区
│   ├─ 某某集团
│   │   ├─ 中行某支行（基本户）✅启用  录入方式：网银导入
│   │   ├─ 建行某支行（一般户）✅启用  录入方式：网银导入
│   │   ├─ 现金账户 ✅启用            录入方式：手动填写

## 六、银行流水解析系统

### 6.1 核心挑战

- 几十家银行，每家导出格式完全不同
- 列名不同（"交易金额" vs "发生额" vs "借方金额"）
- 列数不同、排列顺序不同
- 文件格式不同（Excel、CSV、PDF）
- 日期格式不同（2026/04/03、2026-04-03、20260403、2026年4月3日）
- 金额格式不同（可能含中文大写"壹""贰"）
- 编码不同（GBK、UTF-8）

### 6.2 解析器模板库设计

**核心理念**：银行种类有限（工行/中行/农行/建行等），每种银行的导出格式相对固定。我们要提取的字段是固定的，变的只是银行那边的格式。

```
首次使用：上传样本流水文件 → AI分析文件结构 → 创建"解析器" → 保存到模板库
    ↓
后续使用：同类银行的新账户 → 自动套用已有解析器
    ↓
银行升级网银时 → 更新该银行的解析器
```

**解析器工作流程**：
1. 用户上传流水文件
2. 系统检测文件编码（chardet库自动识别GBK/UTF-8）
3. 读取文件内容
4. 匹配已有解析器（按银行名称匹配）
5. 如果匹配到 → 直接按映射规则解析
6. 如果没匹配到 → AI分析文件 → 生成映射建议 → 用户确认 → 保存为新解析器

### 6.3 AI驱动的解析器创建

AI在分析银行流水样本时：
- 自动识别日期格式（不需要手动写正则）
- 自动识别金额格式（包括中文大写数字）
- 自动识别借贷方向
- 自动生成列名映射关系（"交易发生额(借方)" → "expense"）

用户只需要：
1. 上传一份该银行的流水样本文件
2. 告诉系统"这是哪家银行"
3. AI自动分析并生成映射建议
4. 用户确认映射关系是否正确
5. 确认后自动保存为该银行的解析器模板

### 6.4 导入流程详细设计

**导入界面布局**：
```
┌─────────────────────────────────────────────────┐
│  导入银行流水                                      │
│                                                   │
│  选择账户：[某某集团 - 中行某支行 - 基本户 ▼]       │
│                                                   │
│  ┌───────────────────────────────────────────┐    │
│  │                                           │    │
│  │  拖拽文件到此处，或点击选择文件              │    │
│  │  支持：.xlsx .csv .xls                    │    │
│  │  支持多文件批量上传                         │    │
│  │                                           │    │
│  └───────────────────────────────────────────┘    │
│                                                   │
│  已上传文件：                                      │
│  📄 中国银行_20260403.xlsx  ✅已解析  [查看结果]    │
│  📄 建设银行_20260403.xlsx  ⏳解析中...            │
│                                                   │
└─────────────────────────────────────────────────┘
```

**上传后处理**：
1. 系统读取文件 → 自动检测编码
2. 匹配已有解析器：
   - 匹配成功 → 自动解析 → 显示预览
   - 未匹配 → 弹出AI映射创建向导
3. 显示预览表格供用户核对
4. 用户点"确认写入" → 数据入库

### 6.5 导入安全机制

| 机制 | 说明 |
|------|------|
| 重复导入检测 | 导入前检查是否有相同日期+金额+摘要的记录，提示可能重复 |
| 批次撤销 | 每次导入记录batch_id，支持按批次整体撤销 |
| 原始数据保留 | raw_data字段保存银行导出的原始字段，备核查 |
| 原子写入 | 数据库事务保证——要么全写入要么全不写入 |
| 进度反馈 | 大文件上传时显示进度条，后台异步处理 |

### 8.3 资金日报的日期范围

支持灵活的日期范围选择：
- **按日**：选择某一天
- **按周**：选择一周范围
- **按月**：选择一个月
- **按年**：选择一年
- **自定义区间**：任意起止日期
- **补录场景**：出纳出差几天没做日报，回来后选择"4月1日至4月3日"，把几天的流水合在一起做日报

**日报生成逻辑**：
1. 取日期范围起点之前的期初余额
2. 拉取该日期范围内的所有交易流水（包括导入的和手动填写的）
3. 按时间顺序排列，每笔交易保留真实日期
4. 期末余额 = 期初余额 + 该范围所有收入 - 所有支出
5. 日报标题显示"2026年4月1日 ~ 4月3日 资金日报"
6. 每笔交易保留真实日期，但整份日报是连贯的，不割裂

## 九、AI Agent架构设计

### 9.1 设计理念

借鉴 OpenClaw 的 workspace 隔离机制，每个 Agent 有自己的独立工作区，拥有独立的人格、记忆、工具权限和工作流程，但可以共享公共知识。

**核心原则**：
- 每个 Agent 只能看到自己 TOOLS.md 里列出的数据
- 共享记忆在 shared/ 里，所有 Agent 都能读
- 私有记忆在自己的 MEMORY.md 里，其他 Agent 看不到
- 上下文完全隔离，出纳Agent的对话不会污染费用Agent的对话
- 按需加载，不一次性全读

### 9.2 Agent工作区结构

```
agents/
├── shared/                       ← 公共区域（所有Agent可读）
│   ├── COMPANY.md                ← 公司基本信息（系统自动从数据库生成）
│   ├── KNOWLEDGE.md              ← 公共知识库（总管Agent和用户写入）
│   └── DATA_SCHEMA.md            ← 数据库结构说明（系统自动生成）
│
├── manager-agent/                ← 财务总监（总管Agent，最高权限）
│   ├── SOUL.md / AGENT.md / USER.md / MEMORY.md / TOOLS.md / RULES.md / WORKFLOW.md
│
├── cashier-agent/                ← 出纳助手
│   └── SOUL.md / AGENT.md / USER.md / MEMORY.md / TOOLS.md / RULES.md / WORKFLOW.md
│
├── expense-agent/                ← 费用会计助手
│   └── SOUL.md / AGENT.md / USER.md / MEMORY.md / TOOLS.md / RULES.md / WORKFLOW.md
│
└── custom/                       ← 远期开放：用户自建Agent
```

### 9.3 七个人格文件说明

| 文件 | 作用 | 谁写 | 加载时机 |
|------|------|------|---------|
| SOUL.md | Agent的灵魂/性格定义 | 用户或AI辅助 | 启动时 |
| AGENT.md | 角色定义和身份 | 用户或AI辅助 | 启动时 |
| USER.md | 用户对这个Agent的偏好 | 用户通过管理界面编辑 | 启动时 |
| MEMORY.md | Agent自己的记忆，随使用自动积累 | 系统自动追加 | 需要记忆时加载最近N条 |
| TOOLS.md | 能用什么工具、能看什么数据 | 系统根据UI配置自动生成 | 处理数据时加载 |
| RULES.md | 业务规则 | 用户或AI辅助 | 处理数据时加载 |
| WORKFLOW.md | 标准工作流程 | 用户或AI辅助 | 执行任务时加载 |

### 9.4 按需加载机制

Agent启动时只加载身份信息（SOUL.md + AGENT.md），其他文件按需加载：
- 要处理数据时 → 加载 RULES.md + TOOLS.md
- 需要记忆时 → 加载 MEMORY.md 中最近的N条，不是全部
- 需要公司信息时 → 加载 shared/COMPANY.md

### 9.5 权限控制

权限不硬编码，通过管理界面配置：
- 管理界面上勾选"这个Agent可以访问哪些数据"
- 系统自动更新 TOOLS.md

### 9.6 shared/公共区域数据来源

| 文件 | 来源 |
|------|------|
| COMPANY.md | 系统自动从数据库提取法人单位、大区、账户信息自动更新 |
| KNOWLEDGE.md | 总管Agent和用户可通过管理界面写入 |
| DATA_SCHEMA.md | 系统自动生成，反映当前数据库结构 |

### 9.7 总管Agent（财务总监）

- 最高权限，可访问所有数据
- 可操作shared/公共区域文件
- 可给其他Agent分配任务
- 角色：财务经理/财务总监

### 9.8 Agent阶段实现规划

| 阶段 | Agent能力 |
|------|----------|
| 当前范围 | 一个基础Agent（流水解析助手）+ Agent引擎骨架 |
| 后续范围 | 增加"发票识别Agent"，开放管理界面查看编辑 |
| 远期范围 | 用户可自建Agent，定义角色和工具权限 |
| 远期 | Agent之间可协作，自然语言查询数据库 |

### 9.9 Memory检索策略

| 阶段 | 策略 |
|------|------|
| 当前范围 | 简单文本检索，加载最近N条记忆 |
| 后续 | 引入embedding模型做语义RAG检索 |

## 十一、发票识别系统（后续模块）

### 11.1 OCR技术方案

- 使用百度飞桨 PaddleOCR 作为Python包直接集成到后端
- 不依赖外部工具，不调用Claude Code的技能
- 支持图片（PNG/JPG）和PDF格式

### 11.2 发票类型覆盖

| 类型 | 常见发票 |
|------|---------|
| 费用类 | 办公用品、餐饮费、住宿费、烟酒、小车过路费、工具车过路费、物资采购 |
| 交通类 | 火车票、机票、出租车票 |
| 建筑类 | 机械租赁费发票、建筑服务发票、劳务发票 |
| 会议类 | 会议费发票 |
| 其他 | 任何新增类型的发票 |

### 11.3 发票识别流程

```
上传发票图片/PDF
    ↓
PaddleOCR识别文字内容
    ↓
AI根据发票提取规则提取关键信息
    ↓
按费用类别自动分类
    ↓
显示提取结果供用户确认
    ↓
确认后写入费用记录（关联到交易流水）

### 11.4 发票提取规则模板

- 每种发票类型有一套提取规则

- 后续同类型发票自动套用
- 用户可手动调整规则

### 11.5 批量处理能力

- 支持一次上传多张不同类型的发票
- 系统自动分类识别
- 批量确认写入
- 保证每条信息的正确性

## 十二、工程保障措施

### 12.1 数据安全

| 措施 | 说明 |
|------|------|
| 启动密码 | 可选的PIN码保护，防止他人打开看到财务数据 |
| 数据加密 | SQLite加密扩展（sqlcipher）或应用层加密敏感字段 |
| API密钥加密 | AI配置中的API Key加密存储 |

### 12.2 数据备份与恢复

| 功能 | 说明 |
|------|------|
| 一键备份 | 自动打包数据库+配置+Agent记忆为zip文件 |
| 数据恢复 | 从备份zip恢复 |
| 备份位置 | 用户可选备份存储位置 |

### 12.3 操作审计

| 措施 | 说明 |
|------|------|
| 操作日志 | 关键操作记录到operation_logs表 |
| 批次追踪 | 每次导入分配batch_id，可追溯 |
| 原始数据 | raw_data字段保留银行导出原始数据 |

### 12.4 容错机制

| 场景 | 处理方式 |
|------|---------|
| 导入错误 | 按batch_id整体撤销导入 |
| 重复导入 | 导入前检查相同日期+金额+摘要，提示可能重复 |
| 断电/崩溃 | 数据库事务保证原子性——要么全写入要么全不写入 |
| 浏览器崩溃 | 手动录入时每30秒自动暂存到localStorage |
| AI调用失败 | 优雅降级，提示"AI暂不可用，请手动操作" |
| 文件编码 | chardet库自动检测GBK/UTF-8编码 |
| 端口冲突 | 默认19527端口，被占用自动递增尝试 |
| 多标签页 | 数据库事务锁保护，提示"数据已被其他窗口修改" |

### 12.5 打印支持

- 资金日报页面提供"打印"按钮
- CSS适配打印格式
- 打印时隐藏筛选栏等操作区域
- 纸面布局适配A4纸
