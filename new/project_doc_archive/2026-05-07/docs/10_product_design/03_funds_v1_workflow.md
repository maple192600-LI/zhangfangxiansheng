# 资金板块 V1 主链路与页面状态流

> 状态：needs-rewrite。本文仅作历史主链路参考，当前银行/手工/报表链路以 `../20_execution/12_bank_import_execution.md`、`../20_execution/13_manual_flow_execution.md`、`../20_execution/14_base_data_and_report_execution.md` 为准。

## 一、文档目的

本文档用于把资金板块第一阶段 MVP 的主业务链路、页面职责、状态流转和结果生成路径固定下来。

它直接服务于前端实现、后端服务层实现和后续联调测试。

## 二、V1 目标

V1 只完成资金板块的最小可用闭环。

这个闭环必须让用户完成下面这些动作：

- 维护主数据
- 导入网银流水
- 录入或上传手工流水
- 查看上传结果预览
- 进入手动维护处理异常
- 生成基础数据表
- 生成基础资金结果表
- 进行筛选、导出、打印
- 留痕、备份、回滚

## 三、资金板块主链路总览

当前主链路固定为：

主数据与账户初始化  
到  
网银导入 / 手工流水接入  
到  
上传结果预览  
到  
手动维护  
到  
基础数据表  
到  
现金日记账 / 账户余额表 / 收入明细表 / 支出明细表  
到  
主要账户余额表与基础综合汇总

## 四、主数据阶段

### 1. 输入

- 法人主体
- 资金载体
- 银行账户
- 手工账户
- 现金账户
- 账户别名
- 账户编码
- 法人编码
- 账户状态
- 期初余额
- 余额日期

### 2. 输出

- 可被网银导入和手工流水识别命中的主数据底座
- 别名字典与识别键

## 五、网银导入阶段

### 1. 页面

资金板块 / 工作台 / 网银导入

### 2. 输入

- 银行流水文件
- 已有解析器模板
- 主数据命中规则

### 3. 处理

- 读取文件
- 匹配模板
- 首次识别或复用模板
- 标准化预览
- 生成待处理批次

### 4. 输出

- 网银流水待处理结果
- 批次号
- 原始数据留痕
- 待进入上传结果预览的候选记录

## 六、手工流水阶段

### 1. 页面

资金板块 / 工作台 / 手工流水

### 2. 输入方式

当前固定两种：

#### 系统内快速录入

适合少量手工记录。

#### Excel 手工总表上传

适合多法人、多账户、多现金或无网银账户同时录入。

### 3. 处理

- 系统内输入或上传 Excel 手工总表
- 命中字段方案
- 命中主数据别名
- 生成待处理批次
- 写入预览层，不直接进入正式结果层

### 4. 输出

- 手工流水待处理批次
- 待确认记录
- 待进入上传结果预览的候选记录

## 七、上传结果预览阶段

### 1. 页面

资金板块 / 工作台 / 上传结果预览

### 2. 作用

上传结果预览是所有接入数据进入正式基础数据层前的统一缓冲区。

### 3. 页面中至少应看到

- 待导入文件数
- 待处理项数量
- 已处理项数量
- 待确认项数量
- 重复疑似数量
- 当前批次版本信息

### 4. 用户动作

- 查看识别结果
- 查看待确认项
- 删除明显脏数据
- 跳转手动维护
- 生成基础数据表

## 八、手动维护阶段

### 1. 页面

资金板块 / 工作台 / 手动维护

### 2. 作用

承接所有需要人工处理的数据。

### 3. 当前需要承接的典型问题

- 归属待确认
- 摘要待修正
- 金额异常
- 重复疑似
- 关键字段缺失
- 手工流水与后续导入疑似重复
- 别名字典未命中

### 4. 输出

- 修正后的待处理结果
- 可进入基础数据表的标准化结果
- 被拦截并留在异常状态的记录

## 九、基础数据表阶段

### 1. 页面

资金板块 / 资金日报表 / 基础数据表

### 2. 角色

基础数据表是资金板块统一的基础结果层。

### 3. 来源

- 网银导入处理后的标准记录
- 手工流水处理后的标准记录
- 手动维护修正后的标准记录

### 4. 作用

- 为现金日记账提供底层
- 为账户余额表提供底层
- 为收入明细表、支出明细表提供底层
- 为后续日报重算和汇总提供底层

### 5. 页面能力

- 筛选
- 搜索
- 关键状态查看
- 导出
- 保存修正
- 跳转来源追溯

## 十、结果表阶段

### 1. 现金日记账

从基础数据表中抽取现金类结果视图。

### 2. 账户余额表

按账户展示期初、发生额、期末余额结果。

### 3. 收入明细表

展示收入方向明细。

### 4. 支出明细表

展示支出方向明细。

这些表应全部从统一底层生成，不应另维护多套来源。

## 十一、综合报表阶段

综合报表当前主要承担基础汇总视图。

第一阶段应先接住以下几类：

- 主要账户余额表
- 月末盘点表
- 资金周报
- 资金月报
- 资金年报

第一阶段重点在结构稳定和数据来源统一，复杂分析可后置。

## 十二、状态流转设计

### 1. 数据层状态建议

每条候选基础记录至少应具备以下状态之一：

- 待识别
- 待确认
- 待修正
- 已标准化
- 已进入基础数据表
- 已拦截
- 已回滚

### 2. 批次层状态建议

每个批次至少应具备以下状态之一：

- 已上传
- 处理中
- 待确认
- 部分完成
- 已完成
- 已回滚

## 十三、回滚与重算原则

### 1. 批次回滚

按批次号回滚。

### 2. 结果重算

基础数据表变化后，相关结果表应允许重新生成或重算。

### 3. 异常留痕

即使回滚或修正，原始数据和操作日志也应保留。

## 十四、系统设置与主链路的关系

系统设置中的以下内容直接服务资金板块主链路：

- 账户数据管理
- 报表模板管理
- 银行流水规则
- 手工字段池
- AI 配置
- 备份恢复
- 操作日志

## 十五、第一阶段的重点页面顺序建议

建议前端和后端按下面顺序做透：

1. 账户数据管理
2. 网银导入
3. 手工流水
4. 上传结果预览
5. 手动维护
6. 基础数据表
7. 账户余额表与现金日记账
8. 首页工作总览和待办追踪
9. 备份恢复与日志

## 十六、测试场景建议

### 场景 1

网银导入一批已识别模板文件，成功进入上传结果预览，再进入基础数据表。

### 场景 2

上传一份多主体手工总表，部分记录自动命中，部分记录进入待确认，再通过手动维护进入基础数据表。

### 场景 3

基础数据表生成后，同步查看账户余额表和收入支出明细表。

### 场景 4

某批次存在错误，执行回滚后，结果层重算。

## 十七、结论

资金板块当前主链路已经明确。

后续开发中，任何页面、接口、服务层、数据库设计，都应围绕这条链路展开，不应跳开主链路各自为战。
