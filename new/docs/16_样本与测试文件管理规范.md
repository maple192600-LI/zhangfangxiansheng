# 16_样本与测试文件管理规范

## READ WHEN

接收用户提供的 Excel、模板、报表、银行流水、OCR 图片、导出文件、预期结果文件，或新增 / 修改 / 删除测试样本时必须读取。

## PURPOSE

测试样本用于复现、回归和浏览器验收。临时文件用于一次性调试。两者必须分开放。

用户主动提供并确认用于测试的文件，可以进入样本目录；AI 运行过程中生成的临时产物不得混入样本目录。

## DIRECTORY

测试样本放：

```text
dev/fixtures/
  input/
  expected/
  variants/
  unknown/
  regression/
  browser/
```

临时产物放：

```text
data/tmp/
data/downloads/
```

## SAMPLE TYPES

| 类型 | 目录 | 用途 |
|---|---|---|
| 原始输入样本 | `dev/fixtures/input/` | 已知模板、正常输入 |
| 预期输出样本 | `dev/fixtures/expected/` | 对照输出、报表结果 |
| 变体样本 | `dev/fixtures/variants/` | 同类格式变化 |
| 未知样本 | `dev/fixtures/unknown/` | 触发学习或适配流程 |
| 回归样本 | `dev/fixtures/regression/` | 防止已修复问题复发 |
| 浏览器验收样本 | `dev/fixtures/browser/` | 前端上传和操作验收 |

## SAMPLE MANIFEST

每个样本目录必须有清单：

```text
dev/fixtures/manifest.md
```

清单字段：

```markdown
| 文件 | 类型 | 用途 | 关联任务 | 预期行为 | 是否可删除 |
|---|---|---|---|---|---|
```

## MUST

- 用户提供并确认用于测试的样本，放入 `dev/fixtures/`。
- AI 生成的导出文件、截图、下载文件，先放入 `data/`。
- 只有被确认需要长期回归的文件，才能从 `data/` 移入 `dev/fixtures/`。
- 新增样本必须更新 `manifest.md`。
- 修改预期输出必须说明原因。
- 删除样本必须确认它没有被测试、浏览器验收、规则、工作流引用。

## NEVER

- 不要把样本放进 `app/`。
- 不要把临时导出当成预期输出样本。
- 不要为了让测试通过而修改 `expected/`。
- 不要只保留一个样本覆盖所有场景。
- 不要用文件名驱动业务逻辑。
- 不要删除用户提供的样本而不说明。

## EXPECTED OUTPUT RULE

修改 `expected/` 前必须回答：

```text
为什么预期结果要变？
是需求变化，还是代码错误？
哪些测试受影响？
用户是否确认？
```

如果只是当前代码输出不一致，不得修改 `expected/` 掩盖问题。

## DELETE SAMPLE RULE

删除样本前必须检查：

```text
测试引用
浏览器验收引用
规则引用
工作流引用
PR 记录引用
manifest.md
```

删除后必须更新 `manifest.md`。

## DONE

涉及样本的任务完成时必须输出：

```text
新增样本：
修改样本：
删除样本：
manifest 更新：
临时产物位置：
是否进入提交：
```
