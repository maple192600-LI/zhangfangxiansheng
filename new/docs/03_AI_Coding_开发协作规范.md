# 03_AI_Coding_开发协作规范

## 0. 开发前强制流程

每个开发任务开始前必须执行：

1. 先读本文件。
2. 根据“任务路由表”读取对应规则卡。
3. 复述目标、修改范围、验收路径。
4. 再开始改文件。

如果缺少必须读取的文档，停止并报告，不要猜。

---

## 1. 全局硬规则

MUST:

- 先理解目标，再写代码。
- 只改与本任务直接相关的文件。
- 新增文件必须放入文件地图定义的目录。
- 实现类任务不得直接在 `main` 开发。
- 用户可见功能必须做浏览器真实操作验收。
- 模板、规则、工作流、Skill 必须版本化。
- 账户、组织、部门、项目、客户、供应商等主数据必须使用稳定 `id` 关联。
- 替换实现时必须删除原入口、原引用和原文件。
- 开发阶段用户说删除时，默认彻底删除全链路，不是隐藏或只删前端。
- 正式产品中的删除必须先定义语义：取消、删除草稿、撤销导入、作废、停用、归档、清理、彻底删除。
- 临时文件、下载文件、运行数据库、日志必须放在 `data/` 并在完成前清理或说明。
- 依赖必须安装在项目隔离环境内。
- 完成前必须执行项目护栏检查；护栏未实现时不得声明代码任务完成。
- 开工前必须输出任务单；提交 PR 前必须按 PR 模板填写。
- 完成时必须说明用户如何在前端检查。
- 写实施方案时必须包含真实可行的实现方法，不能只写概念和验收。
- Agent、Skill、工作流、连接器、文件适配、报表生成等实施方案，编写前必须查阅相关参考项目，并在文档中写明参考依据。

NEVER:

- 不要把 Excel 样本、文件名、列号、单元格坐标写死进业务代码。
- 不要只跑后端测试就声明用户功能完成。
- 不要为了通过测试而删除、跳过或放宽测试。
- 不要让 Agent 直接写正式数据库。
- 不要覆盖正式规则、正式工作流或正式模板而不保留原版本。
- 不要用假数据冒充功能完成。
- 不要创建并行替代文件绕过问题。
- 不要让同一功能的两套页面、两套接口、两套服务并存。
- 不要把 `data/`、`.venv/`、`node_modules/`、临时导出、日志提交到 GitHub。
- 不要把测试样本、开发文档、运行数据库放进 `app/`。
- 不要在开发阶段把“删除”理解成只隐藏菜单、只删页面或只删列表。
- 不要在正式产品中把“删除”做成只移除前端列表项。
- 不要写只有页面、表结构、提示词和验收标准的实施方案。
- 不要在未研究参考项目的情况下编写 Agent、Skill、工作流、连接器类实施方案。

## 3.1 实施方案写作要求

实施方案必须包含：

```text
REFERENCE BASIS
ARCHITECTURE
MODULE PATHS
CORE INTERFACES
FLOW
DATA MODEL
API
PERMISSION
TESTS
BROWSER ACCEPTANCE
DO NOT
```

`REFERENCE BASIS` 必须写清：

```text
参考项目：
参考能力：
本产品采用方式：
不采用的部分：
```

如果某项能力没有研究依据，必须写明原因和替代验证方式。

```text

```

## 5. 最低完成定义

用户可见功能同时满足以下条件才算完成：

- PRD 或设计文档中有对应需求。
- 数据模型或接口契约有对应定义。
- 前端页面能真实操作。
- 后端逻辑真实生效。
- 规则、映射、工作流没有硬编码。
- 主数据修改后相关页面能联动，或给出受影响提示。
- 被替换的原入口、原引用、原文件已删除。
- 临时文件和运行产物未进入提交。
- 依赖和数据库在项目隔离环境内。
- 项目护栏检查通过。
- 后端自检通过。
- 浏览器真实操作验收通过。
- 有用户可复查的操作路径或截图记录。
