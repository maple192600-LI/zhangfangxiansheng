# 06_问题修复与代码替换规范

## READ WHEN

修 bug、替换实现、重构、接口调整、页面调整、解析器修复、工作流修复前必须读取。

## CORE RULE

替换就是删除原实现。不得让同一功能的两套代码、接口、页面、服务、工作流入口并存。

业务规则和工作流的版本记录可以作为产品数据保留；代码实现、页面入口、API 路由、服务函数不得两套实现并存。

## FIX FLOW

```text
1. 复现问题
2. 定位真实出错链路
3. 列出受影响页面 / API / 服务 / 数据表 / 工作流 / 测试
4. 修改原实现
5. 同步修改所有调用方
6. 删除被替换的原入口和原文件
7. 搜索原引用，结果必须为 0
8. 跑后端自检
9. 跑浏览器验收
10. 更新完成说明
```

## NEVER

- 不要创建并行替代文件绕过原问题。
- 不要新增一个相似接口，让原接口继续留着。
- 不要新增一个相似页面，让原页面继续出现在路由或菜单里。
- 不要只改当前报错点，不查调用链。
- 不要降低校验、跳过异常、删除失败测试来掩盖问题。
- 不要把失败实现留在主分支。

## GLOBAL IMPACT CHECK

修改前必须回答：

```text
改这里会影响哪些页面？
改这里会影响哪些 API？
改这里会影响哪些数据库字段或查询？
改这里会影响哪些工作流 / Skill / 规则？
哪些原入口必须删除？
哪些测试必须同步修改？
```

## DELETE CHECK

替换完成前必须确认：

- 原文件已删除。
- 原路由已删除。
- 原菜单入口已删除。
- 原 API 调用已删除。
- 原测试已删除或改为验证替代实现。
- 搜索原名称、原路径、原接口无残留。
