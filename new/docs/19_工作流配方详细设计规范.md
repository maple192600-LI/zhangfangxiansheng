# 19_工作流配方详细设计规范

## READ WHEN

设计或实现工作流中心、步骤卡片、画布节点、工作流草稿、样本试跑、启用、停用、回滚、执行记录、节点注册时必须读取。

## CORE RULE

工作流配方是可执行结构化数据，不是静态流程图，也不是散落在按钮事件里的临时代码。

Agent 负责理解意图和生成草稿；工具层负责确定性执行；用户负责确认启用。

已启用工作流再次运行时，不依赖 Agent 再次推理。

## REFERENCE BASIS

| 参考项目 | 借鉴点 | 本产品实现方式 |
|---|---|---|
| n8n | 节点、连线、执行记录、失败节点定位 | 配方 schema、节点注册表、执行事件 |
| ComfyUI | 画布节点可视化、参数可见 | 高级画布模式 |
| Claude Code | 用户描述任务后由 Agent 生成计划并调用工具 | Agent 生成工作流草稿，工具层试跑 |
| pi-mono | 工具事件、上下文压缩、artifact 预览 | 节点事件、输出预览、长流程摘要 |
| OpenClaw | Skill 与工作区能力组合 | Skill 节点注册和工作区内配方 |

不采用：

```text
不要求普通用户手工拖节点
不把画布当唯一工作入口
不让 Agent 每次临场决定已确认流程
```

## ARCHITECTURE

工作流模块应拆成：

```text
RecipeService
NodeRegistry
RecipeValidator
SampleRunner
WorkflowExecutor
WorkflowEventStore
CanvasSerializer
VersionManager
PermissionGuard
```

调用关系：

```text
Agent / 页面
-> RecipeService 创建草稿
-> NodeRegistry 校验节点类型
-> RecipeValidator 校验输入输出
-> SampleRunner 样本试跑
-> 用户确认
-> VersionManager 启用版本
-> WorkflowExecutor 按版本执行
-> WorkflowEventStore 记录节点事件
```

## LIFECYCLE

工作流生命周期：

```text

sample_running
sample_passed

disabled

```

状态必须保存在后端，不得只保存在前端。

## CREATION FLOW

```text
1. 用户上传文件、模板或描述需求
2. Agent 读取上下文、业务对象、字段字典和可用节点
3. Agent 生成工作流草稿
4. 前端用步骤卡片展示
5. 高级用户可进入画布模式查看节点
6. 工具层样本试跑
7. 用户确认
8. 保存为可启用配方
9. 后续运行由工具层按配方执行
```

## IMPLEMENTATION METHOD

生成工作流草稿时必须执行：

```text
1. 读取业务对象、字段字典、文件指纹、可用 Skill 和节点注册表
2. Agent 只输出草稿 recipe
3. RecipeValidator 做结构校验
4. SampleRunner 使用样本执行每个节点
5. 节点输入输出写入 sample_run
6. 失败节点返回中文原因
7. 用户确认后才启用
8. 启用版本只由 WorkflowExecutor 执行
```

如果 Agent 生成的节点类型不存在，必须进入草稿错误状态，不得自动创建正式节点。

## REVIEW MODES

### 简单模式

普通用户只看到：

```text
输入
处理步骤
校验结果
输出
确认按钮
```

### 步骤卡片模式

每个步骤卡片显示：

```text
步骤名称
输入来源
处理动作
输出结果
异常提示
是否需要确认
```

### 画布模式

画布用于高级编辑：

```text
节点拖拽
连线
参数编辑
输入输出查看
失败节点定位
版本对比
```

普通用户不应被要求使用画布完成工作。

## SAMPLE RUN

启用前必须样本试跑。

试跑记录至少包含：

```yaml

version:
input_files:
input_params:
node_results:
validation_results:
output_preview:
errors:
started_at:
finished_at:
```

试跑失败不得启用。

## EXECUTION RECORD

每次执行必须记录：

```yaml
execution_id:
recipe_id:
version:
trigger:
inputs:
node_events:
outputs:
status:
errors:
started_at:
finished_at:
```

执行失败必须能定位到节点。

## VERSION RULE

- 配方变更必须生成后续版本。
- 已启用版本不得被覆盖。
- 可回滚到已确认版本。
- 回滚也必须生成执行记录。

## PERMISSION RULE

节点权限必须显式声明：

```text
read_file
preview_file
query_data
write_staged_data
commit_confirmed_data
export_file
call_connector
run_sandbox_script
```

Agent 不得绕过节点权限直接写数据。

## NEVER

- 不要把工作流写死在页面按钮里。
- 不要让 Agent 每次临场决定正式报表逻辑。
- 不要跳过样本试跑。
- 不要覆盖已启用版本。
- 不要让普通用户必须学会画布才能完成工作。
- 不要让节点输入输出靠自然语言猜测。
- 不要让执行状态只存在前端内存。

## BROWSER CHECK

工作流功能必须验证：

- 用户能生成工作流草稿。
- 步骤卡片可读。
- 画布可查看节点和连线。
- 样本试跑可执行。
- 失败节点可定位。
- 用户确认后才能启用。
- 已启用配方再次运行不依赖 Agent 推理。
- 版本记录可查看。
- 回滚可执行。
