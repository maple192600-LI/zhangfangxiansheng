# 07_GitHub私有仓库与分支PR规范

## READ WHEN

创建仓库、创建分支、提交代码、创建 PR、合并、删除失败分支、多设备继续开发前必须读取。

## MODEL

```text
GitHub 私有仓库：保存代码、文档、版本记录、分支、PR
本地项目副本：开发、运行、浏览器验收
可选第二设备：需要时从同一私有仓库拉取项目副本继续开发
```

GitHub 负责同步和记录；真实运行和页面验收在本地电脑完成。

## MUST

- 仓库必须是 Private。
- 不要直接在 `main` 开发。
- 每个任务创建独立任务分支。
- 任务完成后创建 PR。
- PR 必须写明修改范围、浏览器验收路径、测试结果、风险。
- 失败任务分支必须删除。
- 删除失败分支前，保留文字结论；不保留失败代码。
- 如果启用第二设备开发，开始前必须先拉取 GitHub 远端当前代码。

## NEVER

- 不要把失败分支合并到 `main`。
- 不要把失败代码留在任务分支里长期占位。
- 不要绕过 PR 直接把任务结果并入 `main`。
- 不要把 `data/`、`.venv/`、`node_modules/`、临时文件推送到 GitHub。
- 不要在多个设备分别改同一任务而不先同步。

## TASK BRANCH FLOW

```text
1. 从 main 拉取远端当前代码
2. 创建任务分支
3. 在本地开发
4. 本地启动产品
5. 浏览器真实验收
6. 清理临时产物
7. 提交 commit
8. 推送到 GitHub
9. 创建 PR
10. 用户确认后合并
11. 删除任务分支
```

## FAILURE FLOW

```text
1. 停止继续叠加修改
2. 记录失败原因和已验证事实
3. 删除失败任务分支
4. 回到 main 当前状态
5. 重新创建任务分支
```

## PR MINIMUM

```markdown
## 目标

## 修改范围

## 删除的原入口

## 浏览器验收

## 临时产物清理

## 风险

## 用户检查路径
```
