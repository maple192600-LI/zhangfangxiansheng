## 1. 产品概述

### 1.1 产品定位

账房先生是一款面向中国财务人员的本地部署财务 Agent 工作台。

产品采用传统财务软件清晰、可理解、可核对的功能页面，同时提供一个可随处唤起的 Agent 工作能力。用户既可以在“文件导入、报表中心、数据中心、规则与自动化”等功能页面中让 Agent 处理所在页面任务，也可以进入独立对话界面，用自然语言直接命令 Agent 完成跨页面、跨文件、跨数据表的工作。

### 1.2 产品愿景

让财务人员拥有一个“会长期记住本单位数据和报表口径的本地财务同事”。

它不替代财务人员做最终判断，也不让 AI 直接自由算账，而是通过 Agent 理解需求、编排工具、生成草稿、沉淀 Skill，再由确定性工具完成解析、计算、入库、导出和校验。

### 1.3 核心产品原则

1. **默认功能页面**：资金、OCR、贷款、预算、规则、异常、备份、日志等作为本产品的默认功能域定义。
2. **双入口 Agent**：业务页面中可以召唤 Agent；独立对话界面也可以直接下命令。
3. **Agent 配置独立管理**：Agent、Skill、记忆、上下文、会话树、压缩摘要、工具权限等只在 Agent 相关界面管理，不混入普通业务工作台或业务目录。
4. **Skill 自由扩展**：Skill 像 Claude Code / Codex / pi 的技能机制一样，可由用户创建、Agent 生成、外部项目打包、未来市场下载，不固定数量。
5. **自动上下文管理**：Agent 必须显示 token、上下文占用、当前模型、当前会话、压缩状态，并具备自动压缩上下文能力。
6. **动态业务对象**：产品不能只写死少数页面和表结构，必须允许用户根据自己的岗位和工作内容创建、导入、维护不同类型的业务数据对象。
7. **确定性执行**：金额计算、汇总、正式写库、正式导出必须由确定性工具完成，Agent 只负责理解、编排、生成草稿和解释。
8. **受治理的文件留存**：上传文件进入受管理的文件仓库，按类型、来源、重要性和用户配置决定留存周期，不无脑永久保存。
9. **本地数据安全**：源数据不被报表过程直接覆盖，关键动作可审计、可回滚。
10. **中文优先交互**：界面、菜单、按钮、提示、错误、帮助说明、Agent 回复和用户命令必须以中文为主；底层工具名、API、数据库字段、脚本函数可以使用英文稳定标识。

| OpenClaw | Agent 工作区、Skill 发现、Skill 按需加载、依赖检查、安全扫描、运行记录 | Agent workspace、全局 Skill 与工作区 Skill、Skill 安装测试、Skill 启用确认、工作区能力治理 |
| Hermes Agent | 记忆、技能、自我改进循环、从执行轨迹中提炼后续改进 | Agent 运行复盘、候选 Skill / 规则 / 工作流草稿、样本评估、用户确认后沉淀 |
| Claude Code | 工具调用循环、文件操作、命令执行、权限确认、MCP 扩展、长任务会话 | 本地文件夹授权、工具执行事件、外部连接器、用户确认、高风险操作拦截 |

### 2.2 功能参考矩阵

| 功能 | 主要参考 | 实现要求 |
|---|---|
| Agent 核心循环 | pi-mono、Claude Code、Claw Code | 用户消息进入 AgentLoop，模型返回 tool_call，权限检查后执行工具，工具结果写事件流，Agent 汇总结果 |
| 工具注册与执行 | pi-mono、Claude Code | 所有工具必须有 schema、权限级别、输入校验、输出摘要和事件记录 |
| 文件夹操作能力 | Claude Code、Claw Code | Agent 只能操作用户授权的文件夹；未授权路径必须被拦截并记录 |
| 权限与路径保护 | pi-mono、Claude Code | 文件、数据库、外部连接器、高风险动作统一走 Permission Guard |
| 会话树与恢复 | pi-mono、Claude Code | 会话可恢复、分支、压缩、查看工具事件，不覆盖原记录 |
| 上下文压缩 | pi-mono、Claude Code | 支持自动压缩、手动压缩、压缩摘要查看和原会话保留 |
| Skill 体系 | OpenClaw、pi-mono、Claude Code Agent Skills | 启动时只加载 Skill 摘要；命中任务时加载完整 Skill；Skill 可带脚本、参考资料、资产和依赖声明 |
| Skill 创建与安装 | OpenClaw、Claude Code Agent Skills | 预置 `skill-creator`；用户或 Agent 可创建 Skill 草稿；启用前必须测试和确认 |
| 自我改进 | Hermes Agent | Agent 从失败记录、重复操作和用户修正中生成候选改进，不自动生效 |
| 外部系统连接 | Claude Code MCP、CLI 工具模式 | 通过 CLI / API / MCP / 连接器统一暴露能力；外部写入必须走权限和审计 |
| 前端工具过程显示 | pi-mono、Claude Code | 用户必须看到工具计划、执行、阻断、失败和结果摘要 |

### 2.3 必须吸收的能力

本项目必须从一开始设计以下能力：

1. Agent 必须有可运行的工具调用循环，不得只是聊天提示词。
2. Agent 必须能在授权范围内操作本地文件夹，并能被权限系统阻断越界操作。
3. Agent 会话可保存、可恢复、可分支、可压缩。
4. Agent 界面显示 token、上下文占用、模型、会话、工具状态和压缩状态。
5. Agent 工具调用过程可视化，并保存可审计事件。
6. Skill 按目录发现，按需加载，支持脚本、参考资料、资产和依赖声明。
7. Skill 支持从外部项目生成草稿，但启用前必须测试和用户确认。
8. Agent 自我改进只能生成候选草稿，不能自动改正式规则、正式 Skill 或正式工作流。
9. 业务页面与独立对话页共用同一套 Agent Runtime。

## 3. 目标用户

### 3.1 核心用户

| 用户 | 主要任务 |
|---|---|
| 出纳 | 银行流水、账户余额、现金日记账、资金日报、资金计划 |
| 会计 | 基础明细、台账、凭证前数据、辅助核算数据 |
| 财务主管 | 异常复核、资金汇总、报表口径确认 |
| 财务 BP / 分析人员 | 多表取数、经营分析、临时报表 |
| 小微企业负责人 | 快速查看资金、收入、支出和异常 |

### 3.2 用户能力假设

用户通常会使用 Excel / WPS，能理解业务口径和报表要求，但不应被要求写代码、SQL、JSON、正则表达式或底层字段映射。

### 3.3 用户对 Agent 的预期

用户希望 Agent 像一个财务同事：

- 能看懂所在页面正在做什么。
- 能读取当前上传的表格和预览结果。
- 能解释为什么某些行异常。
- 能帮用户生成模板、规则、脚本草稿。
- 能把重复工作沉淀成 Skill。
- 能记住本单位常用报表口径。
- 能在长任务中不丢上下文。
- 能让用户随时看到它用了多少 token、还剩多少上下文、是否已经压缩。

## 4. 产品目标

### 4.1 目标

本项目要交付一个可运行的本地财务 Agent 工作台最小闭环。这个闭环的中心是“用户自己的财务业务数据对象可以被导入、识别、建模、入库、展示、查询、调整和复用”。

1. 设计默认功能域、页面体系和导航结构。
2. 提供独立 Agent 对话工作台。
3. 在关键业务页面提供“召唤 Agent”入口。
4. 提供 Agent 配置、Skill 管理、记忆管理、会话管理界面。
5. 支持 token、上下文、模型、会话、压缩状态展示。
6. 支持自动上下文压缩和手动上下文压缩。
7. 支持初始化阶段上传企业基础数据 Excel，由 Agent 识别字段、生成映射草稿、确认后入库，并在基础数据页面展示。
8. 支持用户创建或由 Agent 建议创建新的业务数据对象，例如项目台账、合同台账、保证金台账、预算表、回款计划表。
9. 支持任意工作文件上传后的通用适配流程：识别结构、匹配已有解析器 / 规则 / Skill，未命中时进入 Agent 学习或修正规则流程。
10. 支持 Agent 根据用户意图、文件结构和已有数据对象生成工作流配方草稿。
11. 支持工作流配方的步骤审阅、样本试跑、版本保存和复用。
12. 支持银行流水上传、识别、解析预览、确认入库，作为内置场景之一。
13. 支持报表模板上传与自然语言填报说明，作为内置场景之一。
14. 支持通过确定性工具生成 Excel 报表。
15. 支持文件留存策略、空间占用展示、归档与清理。
16. 支持审计日志、版本记录和关键动作确认。

### 4.2 成功指标

| 指标 | 验收目标 |
|---|---|
| 信息架构 | 默认功能域、页面体系和导航结构清晰，普通财务用户能理解入口含义 |
| Agent 双入口 | 用户可在业务页面召唤 Agent，也可进入独立对话页命令 Agent |
| 上下文状态 | Agent 界面显示 token、上下文占用、模型、会话、压缩状态 |
| 自动压缩 | 长会话接近上下文上限时自动压缩，并可查看摘要 |
| Skill 扩展 | 用户可创建 Skill 草稿，Agent 可从外部项目生成 Skill 草稿 |
| 基础数据初始化 | 用户上传企业基础数据 Excel 后，Agent 能识别字段、生成映射草稿，用户确认后入库并在页面展示 |
| 动态业务对象 | 用户能创建或由 Agent 建议创建新的业务数据对象，而不是只能使用系统写死的几张表 |
| 通用文件适配 | 任意工作表格上传后，系统能匹配已有解析器；未命中或低置信度时 Agent 能创建或修改解析器 / 规则草稿 |
| 工作流配方 | Agent 能把“上传文件 / 查询数据 / 填写模板 / 生成报表”的需求转成可执行、可审阅、可版本化的工作流配方 |
| 可视化工作流 | 普通用户能用步骤卡片审阅流程，高级用户可进入画布模式查看或编辑节点 |
| 规则复用 | 同类文件第二次上传时，可命中已确认规则，无需再次配置 |
| Agent 可配置 | 用户能创建自定义 Agent，分配职责、权限、Skill 和可访问数据范围 |
| 内置场景验证 | 银行流水和报表生成作为内置场景，验证通用适配机制可落地 |
| 存储治理 | 系统能展示文件占用、留存策略、归档状态和可清理文件 |
| 安全审计 | 入库、规则生效、Skill 启用、数据调整、报表生成均有记录 |

## 6. 信息架构与页面设计

### 6.1 总体导航

前端界面应围绕本产品定位设计，优先服务财务人员的日常工作路径、数据核对习惯和报表交付流程。

推荐采用适合财务软件的产品气质：左侧主导航、顶部页签、内容壳、表格优先、克制稳定的视觉风格。

导航设计原则：

- 先按用户日常任务排序，再按系统底层能力排序。
- 首页之后优先出现“今天要做什么”，而不是先出现“数据底座在哪里”。
- Agent、Skill、工作流、连接器是产品核心能力，不藏在系统设置里。
- 数据中心是可查、可维护、可追溯的底座，但不是普通用户每天第一入口。
- 资金、OCR、贷款、预算等作为默认业务域设计，并归入工作台体系。

推荐主导航：

```text
首页
  工作总览 / 待办追踪 / Agent任务 / 异常提醒 / 快捷入口

日常工作台
  资金工作台 / 预算工作台 / 贷款工作台 / OCR工作台 / 自定义工作台

  上传文件 / 解析预览 / 导入批次 / 异常处理 / 模板学习

报表中心
  报表模板 / 报表生成 / 报表记录 / 导出记录

数据中心
  基础数据对象 / 动态业务对象 / 字段字典 / 数据来源 / 数据查询

工作流中心
  工作流配方 / 执行记录 / 可视化画布 / 版本与回滚

Agent 中心
  Agent 对话 / Agent 列表 / Agent 会话树 / Skill 管理 / 记忆管理 / 工具监控

规则与自动化
  解析规则 / 字段映射 / 报表绑定 / 校验规则 / 定时任务

  CLI / API / MCP / ERP-OA / 飞书 / 文件夹监听

文件仓库
  文件列表 / 留存策略 / 归档任务 / 空间占用 / 清理日志

系统管理
  模型配置 / 用户权限 / 安全策略 / 备份恢复 / 操作日志
```

## 7. 前端界面要求

### 7.1 视觉框架

默认视觉方向：

- 柔和浅暖背景
- 低饱和绿色主强调色
- 砂金 / 暖棕辅助色
- 左侧 280px 导航
- 右侧内容 shell
- 顶部页签
- 表格为主，卡片和图表为辅
- 中文 UI
- 温和、清晰、克制的财务软件风格

### 7.1.1 中文交互与命令

产品面向中文财务用户，不能要求用户记英文命令。

用户可见层必须中文优先：

- 页面标题、按钮、表单、提示、错误原因使用中文。
- Agent 回复、工具过程说明、权限确认说明使用中文。
- 命令面板、快捷命令、斜杠命令使用中文。
- 帮助文档和功能解释使用中文。

系统内部仍使用稳定机器标识：

```text
用户输入：/压缩上下文
命令解析：agent.compact
执行工具：agent.compaction.run
页面显示：正在压缩上下文
```

中文命令必须通过命令注册表映射到底层动作，不得直接把中文文案写进业务判断。

命令注册表至少包含：

```text
command_id
中文名称
中文别名
适用页面
所需权限
参数 schema
确认策略
底层 handler
```

## 9. Skill 需求

### 9.1 Skill 定义

Skill 是 Agent 可以发现、读取、调用、组合、生成和维护的能力包。

Skill 不固定数量，不按业务模块硬编码。

Skill 必须由账房先生自己的 Skill 加载与治理机制管理。

Skill 格式兼容 Claude Code / Agent Skills 的 `SKILL.md` 结构，但产品只使用自己的 Skill 目录和治理机制。

Skill 只分两类：

```text
global      全局 Skill，直接放 skills/
workspace   工作区 Skill，放 workspaces/<workspace_id>/skills/
```

如果某个 Skill 只允许部分 Agent 使用，不新增目录，通过 `index.json.allowed_agents` 控制。

一个 Skill 可以包含：

```text
skill-name/
  SKILL.md
  scripts/
  references/
  assets/
  examples/
```

`SKILL.md` 是每个 Skill 包唯一强制要求自带的文件。启用状态、权限、测试记录、绑定关系、工作流节点注册写入产品自己的 `index.json`，不改写导入 Skill 包内部文件。

系统必须从 Anthropic skills 仓库或当前 AI Coding 工具内置来源预装 `skill-creator`，作为全局 Skill：

```text
skills/skill-creator/SKILL.md
```

### 9.2 Skill 来源

Skill 可以来自：

- 系统内置
- 用户手动创建
- Agent 根据用户需求生成草稿
- 本地文件夹导入
- Claude Code / Codex / pi-mono / OpenClaw Skill 兼容目录导入
- GitHub / Gitee / npm / pip 项目打包
- 未来 Skill 市场下载

用户可以要求：

“你去看这个 OCR 项目，把它打包成一个适合回单识别流程使用的 Skill 草稿。”

Agent 需要完成：

1. 阅读项目说明。
2. 判断依赖、入口和许可证风险。
3. 生成 Skill 目录草稿。
4. 写兼容 `SKILL.md` ABI 的 Skill。
5. 写入产品登记表。
6. 生成测试样例。
7. 试运行。
8. 给出风险说明。
9. 等待用户确认启用。

### 9.3 Skill 管理界面

Skill 草稿、Skill 配置和 Skill 文件夹只出现在 Agent 中心或系统管理的模型配置相关界面，不出现在资金工作台或业务报表页面。

Skill 管理器需要支持：

- Skill 列表
- Skill 搜索
- Skill 详情
- Skill 文件树
- Skill 草稿
- Skill 测试记录
- Skill 版本
- Skill 绑定 Agent
- 启用 / 禁用
- 回滚
- 从外部项目创建草稿

### 9.4 Skill 触发方式

Skill 可由以下方式触发：

- Agent 自动判断任务匹配
- 用户在对话中指定
- 页面上下文触发
- 工作流触发
- 定时任务触发
- 其他 Skill 调用

## 10. 工作流、数据与工具层

### 10.1 工作流配方定位

工作流配方是本产品的核心资产之一。

产品不应为每一种财务场景硬编码一套固定流程，而应把“用户上传一个文件、模板或提出一个取数需求，系统如何理解、查询、处理、校验、写入、导出”沉淀为可执行、可审阅、可版本化的工作流配方。

工作流配方不是一张静态流程图，而是一份可以被工具层执行的结构化定义。

典型流程：

```text
输入文件 / 模板 / 数据库 / 外部系统
  -> Agent 理解意图、识别结构、生成草稿
  -> 系统展示可审阅流程
  -> 工具层试跑样本
  -> 用户确认
  -> 保存为可复用配方
  -> 后续同类任务确定性执行
```

### 10.2 工作流交互模式

工作流需要同时服务普通财务用户和高级用户。

推荐三种模式：

| 模式 | 面向用户 | 产品形态 |
|---|---|---|
| 简单模式 | 不想学习节点的普通财务人员 | 上传文件、填写说明、让 Agent 生成流程 |
| 审阅模式 | 大多数业务用户 | 用步骤卡片展示 Agent 生成的流程，用户确认、修改说明、试跑 |
| 画布模式 | 高级用户、实施人员、产品维护者 | 类似 n8n / ComfyUI 的可视化节点画布，可拖拽、连线、改参数、保存版本 |

默认入口应是简单模式和审阅模式。画布模式不应成为普通用户完成工作的前置条件。

### 10.3 工作流节点来源

节点不要求用户手工设计。

节点来源分为三类：

1. **系统内置基础节点**
   - 读取文件
   - 识别表格结构
   - 匹配模板
   - 查询业务数据
   - 筛选数据
   - 汇总数据
   - 字段映射
   - 写入 Excel
   - 校验规则
   - 生成预览
   - 用户确认
   - 保存规则
   - 导出文件
   - 调用连接器
   - 调用 Skill
2. **Skill 注册扩展节点**
   - OCR Skill 可注册“识别回单”
   - 飞书 Skill 可注册“读取飞书表格”
   - ERP 连接器可注册“读取付款申请”
   - Excel Skill 可注册“复杂表格修复 / 格式保真导出”
3. **Agent 组装节点**
   - Agent 根据用户意图、所在页面、上传文件和已有数据对象，自动生成工作流草稿
   - 用户主要负责确认和修正业务口径，而不是手工连线

### 10.4 工作流配方结构

每个工作流配方至少包含：

- 工作流名称
- 适用业务场景
- 输入类型
- 适用模板或文件指纹
- 数据来源
- 节点列表
- 节点参数
- 节点连接关系
- 字段映射
- 筛选条件
- 计算口径
- 校验规则
- 输出位置
- 权限要求
- 版本记录
- 试跑样本
- 用户确认记录

### 10.5 工作流执行原则

- Agent 负责理解、规划、生成草稿、解释和维护。
- 工具层负责确定性执行。
- 用户负责确认业务口径、高风险动作和正式生效。
- 已确认工作流再次运行时，不需要再次让 Agent 推理。
- 工作流变化时生成后续版本，不覆盖原版本。
- 工作流执行过程需要可视化展示，失败步骤可定位、可重试。

### 10.6 本地事务库

建议继续使用 SQLite 保存本地事务数据和系统元数据：

- 用户与配置
- Agent 配置
- Skill 注册信息
- 会话元数据
- 审计日志
- 导入批次
- 主数据
- 动态业务对象定义
- 动态字段定义
- 文件元数据
- 文件留存策略
- 报表模板
- 规则版本
- 工作流配方
- 工作流执行记录
- 节点注册信息
- 用户确认记录

SQLite 适合作为本地事务库和产品状态库，保存“什么是已确认、已启用、可回滚、可审计”的内容。

Agent 不应默认获得任意 SQLite 写权限。对普通任务，应通过受控业务工具访问 SQLite，例如“读取业务对象”“查询导入批次”“查看规则版本”。高级 SQL 查询仅作为受权限控制的调试或专家能力。

### 10.7 本地分析引擎

建议使用 DuckDB 作为本地分析引擎。

DuckDB 用于：

- 大 Excel / CSV 临时分析
- 多表关联
- 报表取数
- 临时报表查询
- 大批量流水汇总
- Agent 试算草稿
- 工作流样本试跑
- 复杂报表取数

DuckDB 不替代 SQLite，也不由前端直接调用。它是后端确定性工具层的一部分。

DuckDB 更适合“算”和“查”，SQLite 更适合“管状态”和“留痕”。两者之间应通过统一的数据访问层隔离，避免前端或 Agent 直接关心到底查询的是 SQLite 还是 DuckDB。

### 10.8 LibreOffice 集成策略

不建议把 LibreOffice 作为系统强依赖，也不建议把它当成主数据库或主报表引擎。

原因：

- 安装体积和部署复杂度较高。
- 不同系统环境下 headless 稳定性需要测试。
- 常规 Excel 读取、写入、格式处理可以优先由 Python 工具链完成。
- 强依赖 LibreOffice 会影响“双击即用”的轻量体验。

但应预留 LibreOffice 作为可选增强工具或 Skill 能力：

- 高保真打开或转换复杂 Excel / Word / PDF
- Excel 到 PDF 的格式保真导出
- 某些复杂表格的渲染预览
- 对外部模板做兼容性检查
- 作为 Excel Skill 的后端工具之一

产品策略：

- 默认不要求用户安装 LibreOffice。
- 如果系统检测到 LibreOffice 或用户启用便携版路径，则开放相关增强能力。
- Excel 基础解析、字段识别、数据入库、报表写入仍优先走确定性 Python 工具链。
- Agent 只能通过受控工具调用 LibreOffice，不直接操控用户文件。

### 10.9 文件与数据分层

```text
文件仓库层
  上传文件、外部同步文件、导出文件和中间预览文件按策略管理

只读源数据层
  解析后的标准化数据

动态业务对象层
  用户自定义基础数据、台账、明细表、业务宽表

调整与规则层
  用户调整、模板、规则、Skill、Artifact

报表输出层
  报表模板、生成记录、导出文件
```

### 10.10 文件留存治理

系统不能把所有上传原始文件无条件永久保存。

文件保存原则：

- 文件内容存放在受管理的文件仓库中，不直接塞进业务数据库。
- 数据库只保存文件元数据，例如文件名、hash、大小、来源、上传时间、业务对象、导入批次、保留策略。
- 文件按 hash 去重，同一文件不重复存储。
- 文件可按类型设置默认留存周期。
- 中间预览文件、临时文件和失败试跑文件应自动清理。
- 高价值原始文件可长期保留。
- 用户可查看空间占用、归档状态和可清理文件。
- 删除或归档文件不应破坏已入库结构化数据、审计记录和规则版本。

默认策略建议：

| 文件类型 | 默认留存策略 |
|---|---|
| 基础数据初始化文件 | 默认保留，可由用户归档 |
| 银行流水原始文件 | 默认保留 12-24 个月，可配置 |
| 用户确认后的报表模板 | 长期保留，版本化 |
| 临时报表模板 | 用户选择保存或 30 天后清理 |
| 中间预览文件 | 7-30 天自动清理 |
| 外部系统同步原始包 | 可只保留同步日志、hash 和抽样文件 |
| 导出报表 | 默认保留最近版本，原版本可归档 |

### 10.11 工具权限

Agent 调用工具必须经过权限控制。

工具类型：

- 读取文件
- 预览 Excel
- 解析 Excel
- 查询业务数据
- 查询 SQLite
- 查询 DuckDB
- 运行沙箱脚本
- 写报表文件
- 读取工作流配方
- 创建工作流草稿
- 试跑工作流
- 创建 Skill 草稿
- 运行 Skill 测试
- 生成审计记录

高风险动作必须确认：

- 正式写库
- 批量调整数据
- 启用 Skill
- 替换正式规则
- 覆盖报表模板
- 创建或修改动态业务对象结构
- 启用或替换正式工作流
- 删除文件

## 11. 核心业务功能需求

### 11.1 基础数据初始化

用户在初始化阶段应先建立自己的业务数据底座。不同岗位需要维护的基础数据不同，系统不能假设所有用户都只需要银行账户。

基础数据中心需要支持：

- 上传企业基础数据 Excel。
- Agent 识别表格用途，例如主体、部门、账户、客户、供应商、项目、合同、员工、科目、收支分类等。
- Agent 识别表头、字段类型、主键、重复列、空值、枚举值和可能的关联字段。
- 如果系统已有对应业务对象，则生成字段映射草稿。
- 如果系统没有对应业务对象，则建议创建新的动态业务对象。
- 工具层试导入样本数据。
- 用户确认字段映射、数据对象和导入策略。
- 数据写入本地事务库。
- 页面展示导入后的数据，支持查询、筛选、修改、导出和查看来源。

初始化成功的核心标准是：用户可以把自己现有的企业基础数据表带入系统，而不是被迫适配系统预设的少数表。

### 11.2 动态业务对象

产品需要支持用户创建自己的业务数据对象。

业务对象示例：

- 项目台账
- 合同台账
- 保证金台账
- 农民工工资台账
- 预算表
- 回款计划表
- 应收明细表
- 应付明细表
- 资金计划表
- 用户自定义经营分析表

每个动态业务对象需要包含：

- 对象名称
- 业务说明
- 字段列表
- 字段类型
- 主键或唯一键
- 可选枚举
- 关联对象
- 导入方案
- 展示视图
- 权限范围
- 版本记录

Agent 可以建议创建业务对象，但创建或修改对象结构必须经过用户确认。

### 11.3 通用文件适配流程

所有上传文件都应先进入统一适配流程，而不是为每一种文件硬编码一套页面逻辑。

统一流程：

1. 上传文件或模板。
2. 文件进入文件仓库并生成元数据和 hash。
3. 系统识别文件类型、表头、结构、来源、指纹。
4. 匹配已有解析器、规则、业务对象或 Skill。
5. 命中且置信度足够时，确定性执行并生成预览。
6. 未命中、低置信度或格式变化时，Agent 进入适配流程。
7. Agent 创建或修改解析器 / 规则 / Skill 草稿。
8. 工具层试跑样本。
9. 用户确认。
10. 保存为后续版本能力资产。
11. 后续同类文件自动复用。

该流程适用于银行流水、基础数据、合同台账、发票台账、预算表、收入表、支出表和用户自己的 Excel 模板。

### 11.4 网银导入

用户在“资金板块 > 工作台 > 网银导入”上传银行流水。

系统需要：

- 按文件留存策略保存或归档原始文件
- 识别文件指纹
- 匹配已有模板
- 解析预览
- 标记异常行
- 页面内可召唤 Agent 解释异常或学习模板
- 用户确认后入库

网银导入是通用文件适配流程在资金场景中的内置实现，不是系统唯一的文件处理能力。

### 11.5 未知模板学习与规则修正

当未命中模板时：

1. 页面提示“未识别到已知模板”。
2. 用户可以点击“让 Agent 学习此模板”。
3. Agent 进入当前文件上下文。
4. Agent 生成字段映射和解析草稿。
5. 工具层试跑样本。
6. 用户确认。
7. 生成可复用模板、解析器、规则或 Skill 草稿。

当已有模板命中但试跑失败，或银行、ERP、OA、用户自定义模板格式发生变化时，系统不应要求开发人员改代码，而应进入“规则修正”流程：

1. 标记原规则版本不适配当前文件。
2. Agent 对比原格式和后续格式。
3. Agent 生成规则修改草稿。
4. 工具层用样本回归测试。
5. 用户确认后保存为后续版本。
6. 原版本保留，可回滚。

### 11.6 基础数据表

基础数据表是用户业务数据底座的一个展示入口，不应只服务资金报表。

要求：

- 支持动态业务对象切换
- 支持筛选
- 支持查看来源文件和导入批次
- 支持查看调整记录
- 支持召唤 Agent 分析异常
- 不允许 Agent 未经确认直接改源数据

### 11.7 报表模板管理

用户可以上传 Excel 模板。

Agent 可以：

- 识别单元格结构
- 识别表头和循环区域
- 根据用户填表说明生成绑定草稿
- 调用工具试生成预览

Skill 草稿和绑定规则的管理仍放在 Agent / Skill 管理界面，报表模板页面只展示业务结果和确认入口。

报表模板变化时，应走与文件适配相同的规则修正机制：原规则不覆盖，后续规则作为草稿试跑，用户确认后成为后续版本。

### 11.8 资金日报表与综合报表

默认提供以下资金报表页面：

- 现金日记账
- 账户余额表
- 收入明细表
- 支出明细表
- 主要账户余额表
- 月末盘点表
- 资金周报
- 资金月报
- 资金年报

这些页面提供：

- 参数选择
- 报表预览
- 导出
- 召唤 Agent 解释或调整草稿

正式报表生成必须由确定性工具执行。

### 11.9 OCR、贷款、预算

提供入口与基础页面结构。

这些模块可作为占位或轻量能力，后续通过 Skill 扩展。

示例：

- OCR 项目可打包成 OCR Skill。
- 贷款台账可由贷款 Agent 或贷款 Skill 扩展。
- 预算资金计划可由预算 Agent 或预算 Skill 扩展。

### 11.10 工作流中心

工作流中心用于管理“系统如何完成一项财务工作”的可复用配方。

工作流中心需要支持：

- 工作流配方列表
- 工作流详情
- 步骤卡片审阅
- 可视化画布模式
- 节点参数查看与编辑
- 样本试跑
- 执行记录
- 版本对比
- 回滚
- 启用 / 停用
- 绑定 Agent、Skill、业务对象、报表模板或文件指纹

用户可以通过以下方式创建工作流：

- 在业务页面上传文件后，由 Agent 自动生成草稿。
- 在独立 Agent 对话页描述需求，由 Agent 生成草稿。
- 在报表模板页面上传模板并提供填表说明，由 Agent 生成草稿。
- 高级用户进入画布模式，基于系统内置节点和 Skill 节点手工编辑。

工作流中心不应要求普通用户学习节点编排。普通用户默认看到的是“步骤卡片 + 预览 + 确认”，高级用户才进入画布。

## 13. 非功能需求

### 13.1 本地部署

产品以本地单机部署为优先。跨平台和多设备继续开发属于可选能力，不作为默认使用前提。

日常使用不要求命令行。

### 13.2 性能

目标：

- 10 万行以内 Excel 可预览和导入。
- 常规报表生成在可接受时间内完成。
- 大数据量查询交给 DuckDB。
- Agent 工具输出过长时自动折叠并可压缩上下文。

### 13.3 隐私

- 默认本地保存数据。
- 调用外部模型前脱敏。

- 模板识别
- Agent 调用
- Skill 启用
- 工具执行
- 用户确认
- 数据入库
- 动态业务对象创建或结构变更
- 报表生成
- 规则变更
- 工作流草稿、试跑、启用、停用和回滚
- 上下文压缩摘要

## 14. 验收标准

### 14.1 Agent 能力验收

1. 用户可以在独立 Agent 对话页发起任务。
2. 用户可以在网银导入页面召唤 Agent。
3. Agent 能读取所在页面上下文。
4. Agent 工具调用过程可见。
5. Agent 状态栏显示 token、context、model、session、压缩状态。
6. 长会话接近上下文上限时自动压缩。
7. 用户可查看压缩摘要。
8. 用户可恢复会话记录或切换分支。

### 14.2 Skill 验收

1. 用户可以看到 Skill 列表。
2. 用户可以创建 Skill 草稿。
3. Agent 可以根据外部项目生成 Skill 草稿。
4. Skill 草稿不会自动启用。
5. Skill 启用必须确认。
6. Skill 有测试记录和版本。

### 14.3 通用业务闭环验收

1. 初始化阶段，用户上传一份企业基础数据 Excel。
2. Agent 能识别表格用途、字段、主键、枚举和关联关系。
3. 系统已有业务对象时，Agent 生成字段映射草稿。
4. 系统没有业务对象时，Agent 建议创建动态业务对象。
5. 用户确认后，工具层写入数据库，并在基础数据页面展示、查询和修改。
6. 用户上传一份已知格式工作文件，系统能命中已有解析器、规则或 Skill，并生成预览。
7. 用户上传一份未知格式或格式已变化的工作文件，Agent 能创建或修改解析器 / 规则 / Skill 草稿。
8. 草稿必须经过样本试跑、用户确认和版本保存后才能生效。
9. 同类文件再次上传时，系统能复用已确认规则。
10. 用户上传报表模板并提供填报说明后，Agent 能生成绑定草稿。
11. 正式报表由确定性工具生成，生成记录可追溯。
12. 文件进入文件仓库并应用留存策略，用户能查看空间占用、归档状态和可清理文件。
13. 文件归档或清理不破坏已入库结构化数据、审计记录和规则版本。

### 14.4 工作流验收

1. 用户上传报表模板并填写自然语言说明后，Agent 能生成工作流配方草稿。
2. 工作流草稿能以步骤卡片展示给普通用户审阅。
3. 高级用户可以进入画布模式查看节点、连接关系和参数。
4. 工作流必须支持样本试跑。
5. 工作流正式启用前必须经过用户确认。
6. 已启用工作流再次运行时，工具层可确定性执行，不依赖 Agent 再次推理。
7. 工作流变更生成后续版本，原版本可回滚。

### 14.5 默认功能域验收

默认功能域必须有明确入口和页面方案：

- 首页
- 资金工作台
- OCR 识别
- 贷款管理
- 预算管理
- Agent 中心
- 工作流中心
- 数据中心
- 文件仓库
- 规则与自动化
- 连接器
- 系统管理

业务页面不承载 Agent 配置和 Skill 草稿管理，只提供上下文召唤和结果确认。

## 16. 结论

账房先生不是把传统财务页面替换成一个聊天框。

它应该是：

> 财务功能页面 + 可随处唤起的上下文 Agent + 独立 Agent 对话工作台 + 可扩展 Skill 系统 + 自动上下文压缩与会话树 + 本地确定性财务工具链。

用户继续在熟悉的页面中处理资金、报表、规则、异常和台账；Agent 则在需要理解、编排、学习、解释和沉淀流程时介入。

这样既兼顾财务软件的确定性和可控性，也吸收了 pi-mono 这类现代 Agent harness 在会话、上下文、Skill 和扩展机制上的优势。
